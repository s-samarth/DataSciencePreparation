Systems & Hardware Foundations

The substrate everything else runs on — concurrency, the memory hierarchy, and how we even measure "fast."

1 · Multithreading vs Multiprocessing

concurrency ≠ parallelism · the GIL is the whole story

A process is a full program with its own memory; a thread is a worker inside a process sharing that memory. Spawning a process is expensive (fresh memory, descriptors); spawning a thread is cheap.

Why Python threading feels "not really parallel"

CPython's GIL (Global Interpreter Lock) ensures only one thread executes Python bytecode at a time per process. So 16 threads on 16 cores still run Python code one-at-a-time — they take turns. But the GIL releases during I/O and many C-extension calls (NumPy, PyTorch, network, disk). That's the entire nuance.

  • Threading wins on I/O-bound work: waiting on API calls, DB, network, disk. While thread A waits on HTTP, thread B runs. This is concurrency.
  • Multiprocessing wins on CPU-bound pure-Python work: separate interpreters, each its own GIL → genuine parallelism. Cost: no shared memory, data passes via pickling (slow), so use chunky work per process.
  • async/await is often best for thousands of concurrent network calls — single thread, switches at every await, lower overhead than threads.

The AI-engineer decision table

BottleneckUse
LLM API calls, DB, networkasync > threading
Reading thousands of filesthreading
CPU-bound Python (rare in ML)multiprocessing
Data loading/preprocessing for trainingmultiprocessing — this is what DataLoader(num_workers=N) is
GPU inferenceneither — the GPU is the parallel engine; you batch
One line

Threading = one interpreter, workers take turns, great for waiting; multiprocessing = many interpreters, true parallelism, great for computing; the GIL is why CPython threading can't give CPU parallelism.

2 · L1 / L2 / L3 Cache & the Memory Hierarchy

every ML perf trick is secretly about keeping data in fast memory

CPUs are absurdly fast; RAM is slow by comparison. Caches exist because moving data is the expensive part of modern computing, not the math.

The desk analogy

  • L1 = desk in front of you (tiny, what you use right now; split L1d data / L1i instructions)
  • L2 = drawer under the desk
  • L3 = shelf shared with roommates (other cores)
  • RAM = warehouse downstairs · SSD = warehouse in another city

Why this matters for AI

  • Blocked/tiled matmul: BLAS splits matrices into tiles that fit in L2/L3, does all work while hot, then moves on → 10–100× over naive loops.
  • Batch size: weights load to cache once, reused across the batch → amortizes the memory load.
  • Flash Attention: keeps attention tiles in GPU SRAM, never materializes the full N×N matrix in slow HBM. Same math, far less memory traffic.
  • Memory-bound inference: decode loads billions of weights from HBM per token → compute units idle. Quantization helps because smaller weights = less to move.

GPU hierarchy maps onto CPU's

CPUGPU equivalent
L1 / shared memorySRAM (~100–200KB per SM)
L2 cacheL2 (~40–50MB on H100)
RAMHBM (~80GB on H100)
Two concepts that unlock everything

Spatial locality — access X, you'll likely access X+1 soon (so caches load 64-byte lines, not single bytes → ordered NumPy iteration is fast, random is slow). Temporal locality — recently used → reused soon (so looping weights over a whole batch is efficient).

One line

L1/L2/L3 are progressively bigger/slower/more-shared layers of fast memory between CPU and slow RAM; every major ML trick (tiling, batching, flash attention, quantization) is about keeping useful data in fast memory longer.

3 · P50 & P99 — Latency Percentiles

averages lie about latency · the tail is real people

P50 = median (half of requests faster). P99 = 99% faster, the slowest 1% are worse. General rule: Pxx means xx% of requests were faster than this value.

Why not the average?

100 requests: 99 take 50ms, one takes 5000ms. Average ≈ 99.5ms (hides the outlier and is dragged by it). P50 = 50ms (typical truth). P99 = 5000ms (the ugly truth). Latency distributions are right-skewed with long tails, so the mean is a bad summary.

Why the tail matters at scale

  • Fan-out compounds: 100 internal calls, each 1% chance of P99-slow → 1 − 0.99¹⁰⁰ ≈ 63% chance at least one is slow. Your user-facing P99 is dominated by internal P99 ("tail at scale", Dean & Barroso).
  • Users feel their request, not your median — 1% bad × 1M/day = 10,000 bad experiences daily.
  • SLAs are written on tail percentiles, never on averages.

Sharp for LLM serving

  • Variable output length → naturally long tail (a 2000-token gen is slow).
  • TTFT (time to first token, prefill+queue) vs TPOT (time per output token, streaming speed) — tracked separately, each as percentiles.
  • Batching interference: one giant request slows others in the batch → throughput optimizations can worsen P99.
  • Cold starts: a request hitting a fresh replica eats the multi-GB model load → pure tail event.
The senior framing

The gap between P50 and P99 measures consistency. P50 100ms / P99 120ms = tight. P50 100ms / P99 3000ms = something nasty lurking in the tail. "How's the latency" is never one number.

One line

P50 = typical experience, P99 = worst-case the users complain about and SLAs are written against; averages hide outliers, tails compound across distributed calls, and LLM tails are structurally large from variable output length + batching.

4 · Throughput vs Latency & Batching

batching raises BOTH throughput and latency

Throughput = requests/tokens completed per unit time (system output rate). Latency = how long one request takes (one user's wait). Batching trades the second for the first.

Why batching trades latency for throughput

The GPU hates being underfed. Loading weights from HBM is the expensive part. Process 1 request → load billions of weights for a few tokens, compute units idle. Process 32 together → load each weight once, reuse across all 32 → ~32× useful work for ~same memory cost. That's the throughput win. But each request now waits (queue + shared slower step). The bus vs taxi: bus moves far more people/hour but your individual trip is slower.

When to use a Batch API (the product)

Submit a large job, provider processes asynchronously within ~24h at ~50% discount. The decision rule is one question: is a human (or latency-sensitive system) waiting on this response right now?

Use Batch APIUse real-time API
Bulk embeddings (1M docs)Chat / copilots
Offline eval / LLM-judgeAgents responding to a person
Dataset labeling, synthetic dataAutocomplete
Tolerates minutes-to-hoursNeeds sub-second to few-sec
One line

Batching raises throughput and latency by amortizing the expensive weight-load across many requests; use a Batch API whenever no one is waiting (bulk embeddings, eval, labeling) for ~50% savings, real-time APIs whenever someone is blocked.

5 · Continuous Batching — Resolving the Paradox

decode is one-token-per-step, so the batch reforms every step

The confusion: "if batching = load weights once + compute in parallel, how can requests leave mid-flight without breaking parallelism?" The resolution is the single most important fact about LLM inference.

The key fact

LLM generation is step-by-step, one token at a time. A 200-token answer = 200 forward passes. A "step" = one forward pass = one next token for every request currently in the batch. Within a step, every request needs the identical operation — one pass through the weights. That's perfectly parallel regardless of how far along each request is.

Static batching (the old way) locks {A,B,C} until all finish — B's slot sits idle for 170 steps while A finishes. Continuous batching lets B leave instantly and D take the slot, keeping the GPU packed with active work every step.

One line

Generation is one-token-per-step, so each step is a fully-parallel single forward pass over whoever's present; continuous batching reshuffles membership between steps (state preserved in each request's KV cache), keeping the GPU packed without ever breaking within-step parallelism.

enabled by → PagedAttention / KV cache

6 · LLM Execution Modes (sync / async / streaming)

streaming is orthogonal · the real problem is binding job life to connection life

"Synchronous" is overloaded — it means both "held-open blocking connection" AND "non-streaming." Streaming is a separate axis. That's the source of confusion.

ModeConnectionDeliveryGood for
Sync request-responseheld open, blocksone final resultfast bounded jobs (<~30s)
Sync streamingheld openincrementalchat — bounded but live tokens
Async (job queue)submit + pollpoll for finallong agent runs, batch
Async streaming (pub/sub)submit + subscribeincrementallong runs + live progress

Why agent runtime is variable

An agent is a loop: think → tool → observe → think → … The number of iterations is data-dependent. Simple query = 1 call = 2s; complex = 15 tool calls + retries = 5min. Unbounded and input-dependent.

Why load balancers kill long sync connections

  • Idle/read timeouts (30–60s default) terminate held-open connections → client gets 504 even though the server is still working.
  • Streaming only partly helps: most LB timeouts are idle timeouts; streaming tokens resets the timer — but a 90s silent "thinking" phase still fires it.
  • Connection exhaustion: each long sync connection ties up a worker slot for the whole job.
  • Deploys/autoscaling drain connections → a 4-min-in agent run dies.
  • Retry storms: timeout → client retries → two doomed 5-min runs.
The fix

Decouple job life from connection life: POST → 202 + job_id (connection closes in ms), worker runs the agent independent of any connection, client polls or subscribes via a reconnectable SSE/WebSocket. With async, the streaming connection is disposable; with sync, the connection is the job.

One line

Sync is fine for bounded short jobs; the moment runtime is unbounded (agentic loops) you must submit→job-id→poll/subscribe, because LB idle/total timeouts, connection exhaustion, deploy drains, and retry storms all kill long held-open connections. Streaming is a UX layer on top of either, not a substitute.

7 · The Inference Optimization Stack

everything derives from prefill (compute-bound) vs decode (memory-bound)

The levers

  • Continuous batching + PagedAttention — pack concurrent requests densely. PagedAttention manages KV cache like OS virtual memory (non-contiguous pages, no fragmentation) — vLLM's signature innovation, the substrate under everything.
  • Prefill/decode disaggregation — run the two phases on separate GPU pools so compute-heavy prefill doesn't stall decode streaming (the interference problem). Transfer KV cache between pools over NVLink/InfiniBand. Lets you scale each independently and even use different hardware. Cheaper middle-ground: chunked prefill (break long prefills into chunks, interleave with decode on the same GPU).
  • Parallelism — two categories:
TypeWhat it doesScope
Data / replicaN independent copies + LB → scales throughputmodel fits on 1 GPU
Tensor (TP)split each layer/matrix across GPUs, all-reduce every layer → fits + speeds modelwithin a node (NVLink)
Pipeline (PP)split by layers into stages, assembly lineacross nodes
Expert (EP)distribute MoE experts across GPUsMoE models
Context / sequencesplit one long sequence across GPUsextreme context length

Decision order: fits on 1 GPU → replicas. Fits on 1 node → TP then replicate. Bigger → TP in-node + PP across nodes. MoE → add EP. Huge context → add sequence parallel. TP/PP/EP make the model fit & fast; replicas make the service handle volume.

How prompt caching fits

Prompt/prefix caching saves & reuses the KV cache of a shared prefix (system prompt, long doc, few-shot, chat history) → skips recomputing prefill for the repeated part. It's a prefill-side optimization: slashes TTFT, frees prefill capacity. vLLM/SGLang do automatic prefix caching (SGLang's RadixAttention = prefix tree). It does not touch decode or change outputs.

One line

Optimization stacks on the prefill(compute)/decode(memory) split: continuous batching + PagedAttention for density, disaggregation (or chunked prefill) to stop prefill stalling decode, parallelism in two flavors (replica for throughput; TP/PP/EP/context to fit & accelerate), and prompt caching as a prefill-side TTFT/cost win.

8 · Why Output Tokens Cost ~5× Input

it's GPU-time per token, traced to prefill vs decode

Input tokens are priced by prefill, output tokens by decode. Input tokens share one parallel forward pass (cheap/token); each output token needs its own full serial forward pass reading all weights (costly/token).

"Are GPUs bad at memory bandwidth?" — No, but you found the bottleneck

An H100 has ~3.3 TB/s HBM bandwidth — staggering. The issue is the compute-to-bandwidth ratio (FLOPs per byte is enormous). Prefill exploits it (many tokens per weight load = high arithmetic intensity). Decode wastes it (load a huge weight to multiply by one token = low arithmetic intensity, compute idle). GPUs aren't bad at bandwidth; decode is structurally bad at using the GPU's compute. Batching partially rescues this by raising arithmetic intensity.

The economics chain

  • GPU-time is the real cost (providers rent by the hour).
  • Output tokens consume far more GPU-seconds each (one full serial pass vs a fraction of a shared pass).
  • Output ties up a slot + KV memory for the whole generation (opportunity cost: fewer concurrent users).
  • Output batches less efficiently than prefill → lower tokens/sec/GPU → higher per-token cost.

Electricity is a real but minor component — the dominant cost is GPU capital + opportunity cost. The 2–8× ratio (varies by model) tracks the GPU-second difference. GQA/MLA and better batching narrow the gap; cached input is even cheaper (skips prefill compute).

One line

Output costs more because input is processed in the compute-efficient parallel prefill (many tokens/pass) while each output token is a full memory-bound decode pass (GPU compute idle, hardware occupied longer); pricing tracks GPU-time-per-token and decode burns far more.

9 · Caching for Consistency: Semantic vs Prompt

semantic caching enforces consistency · prompt caching does NOT

A trap hides in the question: the two cache types attack different problems. Semantic caching helps consistency; prompt caching helps latency/cost only.

Semantic caching — the consistency tool

Keys responses by the query's embedding. New query → embed → vector similarity search → above threshold → return the cached response. "What's your refund policy?" and "How do refunds work?" embed nearby → both get the same stored answer → consistency enforced because paraphrases collapse to one response.

The trap — false cache hits

Loose threshold → wrong answers served confidently. Embeddings are blind to negation ("is X allowed" vs "is X not allowed" sit very close). Tight threshold → low hit rate. The whole game is threshold tuning. Great for FAQ/bounded domains, dangerous for high-stakes specifics.

Prompt caching — NOT a consistency tool

Caches the KV cache of a shared prefix (intermediate computation), not the output. After loading the prefix, the model still samples and decodes fresh → with temp>0, two requests on the same cached prefix can still diverge. It buys latency & cost, nothing about output sameness.

LeverFixes
Exact-match cache (hash full prompt)identical inputs → same output · the only true determinism guarantee
Semantic cachesimilar inputs → same output
temperature=0reduces (not eliminates) variance
Structured output / schemaformat consistency
Subtle point

temp=0 ≠ deterministic on real stacks — batched GPU matmuls aren't associative, MoE routing depends on batch composition → tiny logit diffs flip the argmax. Exact-match caching is the only hard guarantee.

One line

Semantic caching enforces consistency by returning a stored output for similar inputs (watch negation/false-hits, tune threshold); prompt caching only caches prefix KV computation for latency/cost while generation still runs fresh — for true determinism use exact-match caching.

10 · Training Structured Output & Tool Calling

not a new capability — text generation shaped by data + constraints
The reframe

A tool call is just text in a specific format the model emits, which your runtime intercepts and executes. The training problem isn't "teach a new skill" — it's "reliably produce well-formed, schema-valid format-text instead of prose."

StageContribution
PretrainingRaw JSON/code syntax literacy (emerges free from code in corpus)
Special tokens + chat templateStandardized parseable structure: tool-call markers, tool role, result separators
SFT (the core)Format + argument extraction + when-to-call, from synthetic verified traces. Includes negatives (don't over-call), ambiguity, parallel calls, error-recovery. Data is heavily synthetic, filtered for: does JSON parse? args match schema? right function?
RL (esp. verifiable rewards)Sharpens selection, arg accuracy, restraint. Tool correctness is programmatically checkable → clean reward (big reason agentic models improved fast)
Constrained decoding (inference, not training)Guarantees syntactic validity by masking tokens that violate the schema at each step (Outlines/XGrammar, GBNF)
Division of labor

Training makes the model want the right structure & content (which tool, what args); constrained decoding guarantees the output parses. You need both — perfect JSON with wrong args is useless, and right call with broken JSON is useless.

One line

Same next-token engine: pretraining gives JSON literacy, SFT on synthetic verified tool-call traces teaches format + when-to-use, RL with execution-checkable rewards sharpens correctness, and constrained decoding guarantees the output parses. No new architecture.

12 · DSPy — Programming, Not Prompting

treat prompts as optimizable parameters

Not a prompting technique — a framework for programming LLMs. Define your pipeline as modules with input/output signatures (question → answer) and let DSPy compile/optimize the actual prompts against a metric.

How the optimization works

Give it (1) program structure, (2) a small training set, (3) a metric. Optimizers (BootstrapFewShot, MIPRO) search over prompt configurations to maximize the metric:

  • Bootstrap few-shot: run the pipeline on training data, keep high-scoring traces, auto-insert them as demonstrations — learns which examples to include.
  • Optimize instructions: MIPRO uses a model to propose candidate instructions, then evaluates them.
  • Prompt optimization as guided search (not gradient descent — weights stay frozen; it optimizes the text around the model). Mental model: "compiler for prompts."

Popularity (2026): rising in credibility among serious LLM engineers tired of brittle hand-tuned prompts; respected (Stanford NLP); still niche vs LangChain/LlamaIndex name-recognition due to a learning curve. Worth knowing because mentioning it signals you think of prompting as an optimization problem.

One line

DSPy treats prompts as optimizable parameters and auto-compiles few-shot examples + instructions against a metric via guided search; rising among sophisticated builders, still niche vs mainstream tooling.

13 · Vector Distance Metrics

L1→L∞ are one formula with a knob · cosine sits outside it
Minkowski: L_p(x,y) = ( Σ |xᵢ − yᵢ|ᵖ )^(1/p) p=1 → L1 (Manhattan): sum of absolute diffs p=2 → L2 (Euclidean): straight-line, squared penalty p=∞ → L∞ (Chebyshev): max|xᵢ−yᵢ|, only the worst dim

As p rises, you weight the largest coordinate diff more and ignore small ones. Low p spreads attention across all dims; high p focuses on the worst. Cosine is separate — measures angle only, magnitude-blind.

MetricPenalizesUse when
Cosineangle only (magnitude-blind)direction = meaning, magnitude irrelevant — almost all embeddings
L2large deviations (squared)magnitude matters; smooth geometric space
L1all deviations linearlyoutlier robustness; high dimensions
L∞only the worst dimany single dim out of tolerance (constraints)
Dot productalignment + magnitudemagnitude is meaningful (popularity/confidence)
The crucial AI reality

For embeddings, use the metric the model was trained under (usually cosine or dot product) — this overrides taste. On normalized vectors, cosine and L2 give identical rankings (‖a−b‖² = 2 − 2·cos for unit vectors), so the real decision is "do I normalize?" L1/L∞ rarely suit dense neural embeddings. High-D distance concentration is why cosine/normalized-dot wins (angle stays discriminative when raw distances concentrate).

One line

L1/L2/L∞ are one Minkowski formula tuned by p (all-dims → worst-dim); cosine measures direction only. For embeddings use what the model trained under (usually cosine/dot), remember normalized cosine≡L2 ranking, and reach for L1/L∞ in classical ML/constraints.

14 · ANN Indexes — HNSW · Annoy · ScaNN

three families: graph · tree · quantization
HNSWAnnoyScaNN
FamilyGraph (navigable small world)Tree forest (random hyperplanes)Quantization + partition
Recall/latencyBest overallLowerSOTA at scale (esp. MIPS)
MemoryHigh (full graph in RAM)Low-ish (mmap, on-disk)Low (quantized)
UpdatesIncremental OK, deletes degradeImmutable (rebuild)Limited/batch
2026 statusDominant defaultDated/legacyStrong at scale, needs tuning

How they work: HNSW = multi-layer graph, greedily hop toward query, descend layers (skip-list for vectors). Annoy = forest of random-hyperplane trees, traverse + merge leaves; mmap'd so loads instantly & shares across processes. ScaNN = anisotropic quantization (preserves components that matter for inner-product ranking, not raw reconstruction) + partition + rerank.

Decision rule

Default to HNSW. Drop to quantization (ScaNN / IVF-PQ) when vectors won't fit in RAM. Annoy only for static read-only/legacy. At real scale, production composes: graph navigation + quantization compression, not one pure algorithm.

The two "are these the same?" clarifications

  • Annoy vs KD-tree: same tree family, different split. KD-tree = axis-aligned, one tree, dies past ~20 dims. Annoy = random hyperplanes, a forest, redesigned to survive high-D embeddings.
  • ScaNN vs Product Quantization: ScaNN uses the PQ family but optimizes for ranking accuracy (anisotropic) rather than reconstruction accuracy. "Smarter PQ for ranking," wrapped in partition+rerank — not plain PQ.
One line

HNSW (graph) = high-recall RAM-hungry default; Annoy (random-hyperplane tree forest) = simple mmap-friendly lower-recall legacy; ScaNN (anisotropic quantization) = memory-efficient scale/MIPS choice. Default HNSW, switch to quantization when RAM-bound, and production composes the two.

15 · Query Rewriting & Post-Retrieval Best Practices

rewriting is PRE-retrieval · reranking is the real post-retrieval win
Query → [PRE: rewrite/expand/decompose] → Retrieve → [POST: rerank/compress/filter] → LLM

Pre-retrieval (fix the query before it hits the retriever)

  • Conversational rewrite — resolve pronouns into standalone queries ("what about its pricing?" → "Qdrant's pricing"). Non-negotiable for chat — retrieval is stateless.
  • Multi-query — generate paraphrases, retrieve each, merge with Reciprocal Rank Fusion. Highest-ROI for ambiguous queries.
  • Decomposition — break multi-part questions into sub-queries (multi-hop/comparative).
  • HyDE — write a fake answer, embed that (closer to answer-docs than the question).
  • Route by complexity (don't over-rewrite simple queries — latency tax); keep the original as a fallback.

Post-retrieval (what to actually do after)

  • Reranking — the highest-value step. Retrieve wide (top 50–100 via cheap bi-encoder), rerank to narrow (top 3–5) with a cross-encoder that scores (query,doc) jointly for precision. The recall-then-precision funnel.
  • Contextual compression — trim chunks to query-relevant sentences (reduces "lost in the middle").
  • Dedupe + MMR diversity.
  • Ordering — LLMs attend best to start & end → put best chunks at the edges, not buried in the middle.
  • Relevance gating — if top reranked result is below threshold, abstain ("I don't have that") rather than hallucinate.
One line

Rewriting is pre-retrieval (conversational de-referencing mandatory, then route into expand/multi-query/decompose/HyDE); the true post-retrieval wins are retrieve-wide-then-cross-encoder-rerank-narrow, compress/dedupe, edge-ordering to dodge lost-in-the-middle, and relevance-gating to abstain on weak context.

16 · Tabular Data in RAG

naive RAG flattens the 2D grid and chunks rows away from headers

Why naive RAG mangles tables

  • Flattens 2D → 1D: the meaning lives in the grid (cell ↔ column header ↔ row label); linear parsing destroys the association — "120" floats free of "Revenue, Q2, 2025".
  • Chunking splits mid-grid: headers land in chunk A, data in chunk B → headerless, uninterpretable numbers.

Good loading techniques (worst → best)

  • Structure-aware extraction → HTML (complex/merged cells) or Markdown (simple). Use layout parsers (unstructured, LlamaParse, Docling, Azure DI).
  • Keep tables atomic; if splitting, repeat the header row in every sub-chunk.
  • Row-level serialization (highest-leverage for retrieval): turn each row into a self-describing sentence — "In Q2 2025, Revenue was 120." Now the value is glued to its meaning and the embedding captures it.
  • Table summarization / multi-vector: embed a clean summary, store/return the full table.
  • Preserve caption + link to the surrounding narrative.
The deeper fork: retrieval vs computation

(A) Semantic lookup ("Q2 2025 revenue?") → row-serialization solves it, the answer is retrievable. (B) Aggregation/computation ("average across quarters?") → RAG cannot do this reliably; the answer isn't in any chunk. Route to Text-to-SQL: load the table into SQL/DataFrame, have the LLM write a query, let the database do the math. So the real best practice is a router: lookup → vector retrieval; analytical → Text-to-SQL.

One line

Good loaders parse tables structure-aware, keep them atomic with headers repeated in any split, and (key) serialize each row into header-injected sentences so values stay glued to meaning; for aggregation queries don't retrieve at all — route to Text-to-SQL.

17 · BLEU · ROUGE · Beam Search

BLEU = precision (translation) · ROUGE = recall (summarization)

BLEU

Modified n-gram precision (n=1..4, geometric mean). Clipped precision stops "the the the the" gaming (clip count to max in reference). Brevity penalty punishes too-short outputs (its backdoor way of caring about recall).

BLEU = BP × exp( Σ wₙ · log precisionₙ ) denominator = CANDIDATE n-grams (precision)

ROUGE

N-gram recall (denominator = reference n-grams — that single swap is the core BLEU/ROUGE distinction). ROUGE-N = n-gram recall; ROUGE-L = Longest Common Subsequence F-measure. Reports P/R/F1 in practice. Clean framing: BLEU designed precision-first, ROUGE recall-first.

Both do surface n-gram matching → blind to synonyms/paraphrase/meaning → the field moved to BERTScore (embedding cosine) and LLM-as-judge.

Beam search

Turns per-step token distributions into a sequence. Greedy = pick top token each step (myopic). Beam search = keep top k sequences each step, expand all, keep best k, repeat. Greedy = beam with k=1.

  • Catches: k× cost; length bias (longer = more sub-1 probs multiplied → fix with length normalization); blandness (optimizes for high probability → safe/generic).
  • Beam for target-answer tasks (translation/summarization); sampling (temperature/top-k/top-p) for open-ended generation — which is why chat LLMs use sampling, not beam.
One line

BLEU = clipped n-gram precision + brevity penalty (translation); ROUGE = n-gram/LCS recall, F1 in practice (summarization); both surface-level and being replaced by BERTScore/LLM-judge. Beam search keeps top-k sequences for target tasks; sampling for creative generation.

18 · Elo Scoring for Model Comparison

sigmoid expectation + SGD update · tie = 0.5

Elo converts a rating difference into an expected win probability, then nudges ratings by the surprise (actual − expected). Big surprise → big change; expected result → tiny change.

Expected: E_A = 1 / (1 + 10^((R_B − R_A)/400)) every 400 points = 10× the odds · equal ratings = 0.5 Update: R_A_new = R_A + K · (S_A − E_A) S_A: win=1, loss=0, TIE=0.5 · K = learning rate (step size) (S_A − E_A) = the surprise term · Elo is zero-sum

Worked: both at 1500, K=32, A beats B → E_A=0.5, R_A=1516, R_B=1484 (A gains 16, B loses 16). A 1700 model tying a 1300 model loses points (was expected to win, scored only 0.5).

Applying to offline eval (the wrinkles)

Sequential Elo is order-dependent — bad for a fixed test set. Fix: shuffle & average over many orderings (what LMSYS Arena does), or use Bradley-Terry / logistic regression (the order-independent batch MLE of the same model). Elo is the online SGD approximation; Bradley-Terry is the closed-form MLE. With only 2 models, Elo is overkill — just report win-rate + CI; Elo earns its keep with many models. Watch LLM-judge bias (position/length/self-preference) — Elo faithfully propagates it.

One line

Elo = sigmoid expectation on rating gap + one SGD step on the surprise (tie scores 0.5); for offline test sets average over orderings or use Bradley-Terry (order-invariant MLE), and beware judge bias contaminating the win/loss labels.

19 · 2026 Benchmarks & the Gaming Problem

old benchmarks are dead · four distinct gaming mechanisms

The structural shift: benchmarks the industry relied on two years ago are saturated. MMLU went GPT-3 ~35% (2021) → 99% (2026) → measures nothing. GSM8K, HumanEval, basic MMLU are in the graveyard.

What still carries signal

DomainLive benchmarks
Reasoning/knowledgeGPQA Diamond (grad science), HLE (Humanity's Last Exam — built to stay unsaturated, <50%, damning calibration: confident when wrong), ARC-AGI-2/3
CodingSWE-bench Verified (real GitHub issues, ~80% frontier), LiveCodeBench (contamination-resistant via post-cutoff date-stamping)
MathAIME 2025/2026
Human preferenceArena Elo (LMSYS)

Agent benchmarks (distinct category)

  • GAIA — general assistant, multi-step web/file/reasoning (~75% top agents)
  • τ²-bench — tool-agent-user with policy adherence; the reliability benchmark (even top models <50%, pass^8 <25%)
  • OSWorld — real desktop computer use · WebArena — browser · Terminal-Bench — DevOps
  • METR HCAST / Time Horizons — longest task an agent finishes 50% of the time

The four gaming mechanisms (don't conflate)

  • 1 · Contamination — test set leaks into training (removing contaminated GSM8K dropped accuracy up to 13%). Defense: post-cutoff/held-out benchmarks.
  • 2 · Scaffold/harness inflation — same model swings 30+ points by orchestration (Opus 4.5: 80.9% Verified vs 45.9% Pro/SEAL — same model, the harness differs). Vendor numbers aren't directly comparable.
  • 3 · Benchmark design flaws — Berkeley got ~100% on several by exploiting validation infra (public answers on HuggingFace, agent code runs in the evaluator's environment). SWE-bench-Verified: a wrong patch can still pass tests.
  • 4 · Selective reporting — best-of-N, best scaffold, single favorable runs inflate by 5–15 points.
How to read them

Benchmarks are diagnostic tests, not verdicts (a cholesterol test doesn't predict blood pressure). No single benchmark — use a portfolio, and the only one that matters for you is a custom eval of 100–200 cases from your real workload. Extra traps: effective context is only 50–65% of advertised (RULER); single-run success ≠ reliability (pass^k collapse).

One line

Old benchmarks saturated & dead; live ones (GPQA/HLE/SWE-bench Verified/LiveCodeBench + agentic GAIA/τ²/OSWorld) carry signal but are distorted by contamination, scaffold inflation, design flaws, and selective reporting — use public benchmarks to shortlist and a custom multi-run domain eval to decide. Progress is real; the magnitude in any launch is inflated.

Cluster F

Reasoning & Planning

The contested, fast-moving question of whether foundation models can actually plan.

20 · The Planning Capabilities Debate

went from "can't plan" to "can plan with scaffolding + verifiers"

The timeline

  • 2022–23: "LLMs can't plan." Kambhampati/ASU PlanBench/Blocksworld — GPT-4 abysmal, success drops as complexity rises. Framing: "approximate retrieval, not principled reasoning." Self-critique didn't rescue it.
  • 2024: the LLM-Modulo compromise — even skeptics: "LLMs can't plan, but can help planning." Use LLM as idea-generator/translator/critic + an external verifier or classical planner for guarantees.
  • Late 2024–25: reasoning models (o1/o3, R1) shift the ground — skeptics' own re-runs call o1 "a quantum improvement... still far from saturating." This is the jump you sensed; it's real.
  • 2025–26: line moved, gap persists — still lag classical planners on hard combinatorial problems; new long-horizon/spatial benchmarks show failures moved to harder regimes ("systemic deficiency in long-term action representation").
Where consensus sits

(1) Capability is real but degrades with horizon & complexity — the most robust finding. (2) Split: autonomous one-shot planning = weak; plan-with-scaffolding (ReAct, verifiers, replanning) = where wins are. (3) The verifier insight: planning works reliably when there's a sound verifier to check the plan (why coding/math advanced faster — cheap to verify); unreliable when correctness rests on the model's own judgment. (4) Unresolved: "real reasoning vs sophisticated retrieval" is partly empirical, partly definitional.

One line

From "can't plan" to "plans well on short-horizon & verifiable tasks with the right scaffolding," but autonomous long-horizon planning with guarantees is still unsolved — so production agents pair reasoning models with verifiers + replanning loops rather than trusting one-shot plans.

AI Security

21 · Guardrails & Safety Metrics

guardrails are a model-independent pipeline · track BOTH sides of the tension
Input → [INPUT GUARDS] → LLM → [OUTPUT GUARDS] → User ↓ ↓ block/flag/sanitize block/regenerate/redact

Input guards

Prompt-injection/jailbreak detection · PII redaction (Presidio) · topic/scope enforcement · content moderation.

Output guards

Content safety re-check · hallucination/groundedness (often highest-value for factual RAG) · schema validation · PII leakage · brand/business rules.

Implementation (cheap → expensive, run cheap first)

TechniqueFor
Regex/rulesPII patterns, banned terms, format
Small classifierstoxicity, injection, topic
Guard models (Llama Guard, ShieldGemma, NeMo)multi-category safety
LLM-as-judgenuanced groundedness/policy (expensive)

Run in parallel not serial; tier by risk. Streaming dilemma: full-response guards conflict with token streaming → buffered streaming, optimistic+retract, or block-then-release by risk tolerance. Failure policy: block+fallback / regenerate / redact / escalate / degrade / shadow-mode log (deploy new guards monitor-only first).

The metric discipline — both sides

Security and experience are in direct tension. Frame as a confusion matrix on "should this have been blocked?":

Actually harmfulActually benign
BlockedTrue Positive ✓False Positive = false refusal
AllowedFalse Negative = security failureTrue Negative ✓
  • Security (min false neg): attack-success/jailbreak-bypass rate, harmful-output rate, PII-leakage, groundedness-failure.
  • Experience (min false pos): false refusal rate (the UX killer — over-blocking benign requests; the "how do I kill a Python process" type), false-refusal by category, user-reported false blocks.
  • Pick the threshold per domain as a product decision (healthcare → tolerate more false refusals; creative tool → tolerate slightly more risk). Maintain a dual eval set (attacks + benign-borderline); red-team continuously.
One line

Guardrails = model-independent input/output pipeline (rules→classifiers→guard-models→LLM-judge, parallel + tiered, each wired to a failure policy), and the non-negotiable is tracking both security failures (false negatives) AND false refusals (false positives) since tightening one worsens the other — tune the threshold per domain, shadow-deploy first.

22 · Attack Landscape & the Frontier

prompt injection is architecturally unsolved · defenses must be structural
The root

LLMs cannot reliably distinguish trusted instructions from untrusted data — to a model, system prompt, user message, retrieved doc, and tool output are all just tokens. There's no hardware instruction/data boundary. Best-defended models still bypassed ~50% of the time with 10 attempts (Intl AI Safety Report 2026).

The 2026 attack surfaces (OWASP Agentic Top: injection · memory poisoning · tool misuse · supply chain · exfiltration)

  • Indirect prompt injection — malicious instructions hidden in retrieved/processed content (not typed by user). Real: hidden in PR descriptions → RCE in Copilot (CVE-2025-53773, CVSS 9.6); EchoLeak zero-click exfiltration in M365 Copilot; a PDF with white-on-white base64 instructions drove Claude to write to a SCADA system and trip a pump. Hiding: white-on-white, zero-width Unicode, base64, image alt-text, HTML comments.
  • Supply chain — Axios npm (100M weekly downloads) got two poisoned versions deploying a RAT in a 39-min window; a coordinated campaign hit Claude Code/Cursor via 30+ malicious skills; 13.4% of agent skills had critical issues. Agentic supply chains are dynamic (fetch & execute at runtime, no review); a poisoned plugin looks like a feature update. Tool/skill description poisoning = the "agent installs by description" vector you flagged.
  • MCP exploits — 8,000+ MCP servers scanned, many with open admin/no-auth; once wired to tools, injection becomes a code-execution primitive (Semantic Kernel RCE).
  • Memory poisoning · config-file injection (CLAUDE.md, .cursorrules) · secret leakage (3 coding agents leaked secrets via one injection).
Why "use AI to defend AI" fails (Willison)

"You can't solve AI security problems with more AI." Guardrail classifiers are probabilistic (train on yesterday's attacks; attacker needs one bypass). Web security learned this — we use parameterization, sandboxing, least privilege. Structural defenses, not predictive ones. If the model is part of the attack surface, you can't trust it to enforce its own security.

The defensive frontier (architectural)

  • Dual-LLM pattern — Privileged LLM plans & uses tools but never sees untrusted content; Quarantined LLM reads untrusted content but has no tools. Breaks the path injection needs to reach the actor.
  • CaMeL (DeepMind) — current SOTA — custom interpreter tracks data provenance via capability tags, enforces policy on tool calls by trust level (control-flow integrity + access control). Mitigates 67–100% of injections on AgentDojo. Treats data trust as first-class (taint tracking).
  • Constrain the action space — general agents are unsecurable; application-specific agents with intentional capability limits are secure today.

Non-negotiable operational defenses

  • Least privilege at the gateway, not the prompt (the agent should be incapable of the catastrophic action) · human-in-the-loop for high-impact/irreversible actions · sandboxed execution · action-screening vs original intent · behavioral monitoring ("valid auth, wrong action") · token/session binding · pin/scan/inventory every dependency & MCP server · no standing secret access.

The frontier foresight

Attacks evolve along two axes — wider untrusted channels and higher-consequence actions:

  • Trust-graph / consensus poisoning — manufacture reputation (fleets of cross-citing sources) so the AI retrieves & trusts the backdoor; SEO-for-LLMs weaponized.
  • Agent-to-agent injection — multi-agent seams reintroduce everything; injection laundering, confused-deputy, self-propagating prompt-injection worms, cascade failures via poisoned shared memory.
  • Vibe-coding monoculture — shared model mistakes = vulnerability monoculture (one flaw across thousands of apps); slopsquatting (pre-register hallucinated package names); sleeper code trained to look normal to AI review.
  • Cyber-physical — air-gaps dissolved by convenience; multimodal injection via the physical sensorium (adversarial stickers, inaudible audio, QR codes in view — you can't sanitize reality); irreversible/lethal consequences. The highest-stakes near-term frontier.
  • AI-vs-AI speed war — autonomous attack/defense outruns human oversight.
What this means for adoption

AI's value (autonomy + broad access) is its danger — every increment of capability is an increment of attack surface. Risk ≈ untrusted-exposure × consequence × reversibility⁻¹ × oversight-gap. Refuse deployments where high untrusted-exposure + high-consequence + low-reversibility stack faster than you can audit. For vibe-coding: don't deploy what you can't audit; separate prototyping from production; be suspicious of AI-recommended dependencies and AI-suggested security patterns.

One line

The attack surface expanded from "trick the chatbot" to "weaponize the agent's tools, memory & supply chain"; since LLMs structurally can't separate instructions from data, durable defenses are the boring CS fundamentals — least privilege, sandboxing, provenance tracking, and isolating tool-access from untrusted-content (Dual-LLM/CaMeL) — with classifiers as a probabilistic outer layer only. The frontier moves toward agent-to-agent worms and irreversible cyber-physical harm.
