Part I — LangGraph & LangChain Internals

01Pydantic — the runtime contract

Pydantic is a data-validation library. Plain Python type hints (x: int) are decoration — ignored at runtime. Pydantic makes them load-bearing: data crossing a boundary must match the declared shape, or it fails loudly.

from pydantic import BaseModel
class User(BaseModel):
    name: str
    age: int
User(name="Sam", age="25")   # coerced → 25
User(name="Sam", age="old")  # ValidationError

It was never the way LangGraph stored state. You saw it everywhere 10 months ago for two reasons: LangChain's own internals are built on it, and the v1 → v2 migration (v2 core rewritten in Rust) caused churn. That awkwardness is gone now — everything's on v2.

The three places it genuinely belongs

  • Tool / function-call schemas — the cleanest way to describe tool arguments. Field(description=...) text goes straight into the schema the model reads, so it's effectively prompt engineering.
  • Structured output — model.with_structured_output(SomeModel) forces the LLM to return validated data matching your class. Replaces brittle "parse the prose and pray" code.
  • Graph state (optional) — supported, but the community default is TypedDict, not Pydantic.
TypedDictPydantic BaseModel

Docs defaultYesLess common

◆ Compression

Pydantic isn't deprecated and was never the state mechanism. Use it where a runtime contract earns its cost: always for tool schemas and structured output; TypedDict by default for graph state, upgrading to Pydantic only when you want validation guarantees (e.g. multi-author production graphs). The Annotated[...] reducer pattern works with both — it's an orthogonal axis.

02TypedDict — typed hints on a dict

A TypedDict is a regular dict with declared keys and value types. Just type hints on a dictionary — nothing more.

from typing import TypedDict
class State(TypedDict):
    messages: list
    count: int

Checking is compile-time only (IDE / mypy) — autocomplete, typo-catching. At runtime it's an ordinary dict; Python won't stop you putting a string in count. That's exactly why LangGraph picked it as the default state container: it behaves like the dict everything already passes around, but gives a typed contract for free.

◆ Compression

TypedDict = "hints for the developer," enforced by the type-checker, plain dict at runtime. Pydantic = "enforced at runtime."

03How an LLM "runs" Python (it doesn't)

The key insight: the LLM never executes anything. It only emits structured output. The execution is plain Python calling plain Python.

Mechanically: the LLM emits {"name":"search","args":{...}} (just a string). Your runtime has a registry — a plain dict — mapping tool names to real callables:

tools_by_name = {"search": search_fn, "calculator": calc_fn}
result = tools_by_name[call["name"]].invoke(call["args"])

The model chose which function and what arguments — like filling out a form. Your machine ran it.

▲ Sandboxing trap

By default there is no sandbox. If a tool does eval(llm_output) or shell access, it runs with your process's full permissions. Sandboxing (Docker, Firecracker/E2B/Modal, restricted subprocess) is something you wrap inside the tool — it is not a LangGraph feature. The genuinely dangerous case is a code-execution tool (LLM writes Python, you run it); a search/calculator tool only risks bad arguments, not arbitrary execution.

★ Interview line

The model emits a tool name and arguments as structured output; the framework maps that name to a registered Python callable and invokes it with trusted code. The model never executes anything. Sandboxing isn't automatic — it's wrapped around tools that run untrusted or model-generated code.

04Annotated & reducers — the merge strategy

Annotated[ActualType, metadata] bolts extra info onto a type without changing it. The first slot is the real type (what type-checkers see); everything after is metadata a framework reads. Python itself ignores it.

The problem reducers solve: when a node returns an update to a state field, does it replace the old value or merge into it? Default is replace (last write wins). For messages you want append instead. The instruction "how do updates combine for this field" is the reducer — a function (old, new) → merged — and you pin it to the field via Annotated.

from typing import Annotated, TypedDict
from langgraph.graph import add_messages
import operator

class State(TypedDict):
    messages: Annotated[list, add_messages]  # append, handles update-by-id
    items:    Annotated[list, operator.add] # concatenate
    count:    int                          # no reducer → replace

Other common typing imports

  • List/Dict/Tuple/Set — capitalized generics; modern Python allows lowercase list[int].
  • Optional[X] — "X or None" (= Union[X, None]).
  • Union[A,B] — "A or B" (modern: A | B).
  • Literal["a","b"] — restricts to specific values; enum-like, big in tool args & structured output.
  • Callable[[int],str] — "a function taking int, returning str."
  • Any, Sequence, Mapping — escape hatch / abstract list-like / dict-like.
◆ Compression

Annotated[T, meta] = "type T, plus a note for the framework." In LangGraph that note is the reducer — the per-field merge strategy that combines updates during a super-step. No Annotated → field defaults to overwrite. The two you'll write constantly: Annotated (reducers) and Literal (constrained values).

05Why "reducer"? + the operator library

"Reducer" comes from functional programming's reduce — folding many values into one by repeatedly applying a two-arg function (reduce(add,[1,2,3,4]) → 10). LangGraph borrows the name because it folds the stream of updates to a field into one merged value — same (accumulator, new) → accumulator shape.

operator is Python's stdlib exposing built-in operators as functions you can pass around — you can't pass the + symbol, but you can pass operator.add. Common in state: operator.add (concatenate lists / sum numbers — by far the most used), occasionally operator.or_ (merge dicts/sets). For anything else, write your own (old, new) → merged.

06Serialization — why "JSON-compatible state"

Serialize = turn a live in-memory object into a flat representation (usually JSON) you can save, send, and reconstruct. LangGraph cares because checkpointers serialize state every super-step to make it durable, resumable, and time-travelable. A non-serializable value breaks the checkpoint write.

The clean test: "If I save this to a file and load it on a different machine tomorrow, does it still mean anything?" Data describes (a name is true anywhere). Resources do (an open socket is action-in-progress, true only here, now, in this process). JSON carries facts, not action-in-progress.

Serializable (data)NOT serializable (live resource)
strings, numbers, booleans, lists, dictsDB connection / network socket
messages (LangChain has a recipe)open file handle
Pydantic models, datetimesthread / lock / running coroutine
structured tool outputfunction / lambda (JSON has no "code")
LLM client object (wraps HTTP client/config)
▲ The one that bites

Never put the model object in state — it wraps a live HTTP client; it's a tool, not data. Put the model's output in state. Create clients/connections fresh inside nodes when needed.

◆ Compression

"JSON-compatible" is the safe rule; the real rule is "anything with a known serialize/deserialize recipe." A live socket can never have one — there's nothing to write down that brings it back. State = facts the graph gathered, not live machinery.

07Send & dynamic map-reduce

Map-reduce = a fan-out then a fan-in. Map: apply the same op to every item, independently, in parallel (one→many). Reduce: fold all per-item results into one (many→one). Classic: count words per document (map), sum the counts (reduce).

Why LangGraph needed Send: a normal edge hardcodes structure at build time. Map-reduce needs the number of parallel branches decided at runtime — if node A produces 7 subtasks you need 7 copies of B; if 200, then 200. You can't draw that edge ahead of time.

from langgraph.types import Send
def fan_out(state):
    return [Send("process_item", {"item": x}) for x in state["items"]]
    #            ^node to spawn        ^private state for THAT branch

If items has 7 elements → 7 parallel branches; 200 → 200. That's the dynamic part. Each branch is the map; the reducer on the shared field is the reduce.

Send vs Command

Both are control-flow objects you return from a node, but: Send(node, state) = fan-out, "spawn (possibly many) parallel branches each with private state" (usually a list). Command(goto=…, update=…) = "update state and go to that node next" (singular).

★ Interview line

Send is LangGraph's dynamic fan-out primitive — a node spawns a runtime-determined number of parallel branches each with its own state (the map); pairing it with a reducer on a shared field gives the reduce. That's why parallelism width is data-determined at runtime, not fixed at compile time.

08Command vs conditional edges

Two answers to "where do I go next?" — you pick based on who needs to know the routing logic.

Conditional edge — routing lives outside the node, in the graph wiring. Topology stays visible. The default.

def route(state): return "node_b" if state["score"] > 0.5 else "node_c"
graph.add_conditional_edges("node_a", route)

Command — routing lives inside the node, returned with the update.

def node_a(state):
    return Command(update={"score": s},
                   goto="node_b" if s > 0.5 else "node_c")
Use conditional edge when…Use Command when…
routing is separable from the node's workonly the node knows where to go (routing entangled with its work)
you want topology legible in the wiringmulti-agent handoffs — the target is the agent's output
a router can reconstruct the decision from stateyou'd otherwise dump scratch data into state just to route
classic branching / quality gatesnavigating to a parent graph (Command(graph=Command.PARENT)) — edges can't
▲ Why not "together"

For a single decision, pick one — a node with Command(goto=…) shouldn't also have conditional edges, or two mechanisms fight over "what's next." The legitimate mix is at the graph level: different nodes use different mechanisms. Never both on one node's single decision.

★ Interview line

Conditional edges externalize routing into the topology, which keeps control flow legible — the default. Command co-locates a node's state update and its routing, right for multi-agent handoffs and parent-graph navigation, where the next step is the node's own output rather than a separable function of state.

09The Functional API

Same engine, imperative style. @entrypoint = "this function is the runnable workflow." @task = "this is a unit of work" (gets checkpointing/retry/parallelism). Your control flow stays plain Python.

from langgraph.func import entrypoint, task
@task
def step1(x): ...
@entrypoint(checkpointer=saver)
def workflow(inp):
    a = step1(inp).result()
    if a > 0.5:                # plain Python if — no router, no edge
        b = step2(a).result()
    return b
Functional API wins when…Graph API wins when…
complex nested if/for/while logictopology is the valuable, inspectable thing
retrofitting existing pipeline codegenuine parallel fan-out
structure is trivially linear, logic is gnarlycomplex multi-agent topologies
messy data-dependent iterationhuman-in-the-loop interrupts at named nodes; visualization
◆ Compression

Functional API = LangGraph's durability/checkpointing/streaming over ordinary imperative Python, via @entrypoint + @task. "Rarely used" because if you reach for LangGraph you usually want what graphs give: visible topology, parallelism, interrupts, multi-agent. It's "LangGraph's runtime without LangGraph's graph."

10Tool error handling — and where results live

Why tool output "looks far from state": the tool and the node are different layers. A tool returns a naked value and knows nothing about state. The ToolNode is the adapter that wraps that value into a ToolMessage and appends it to state["messages"] via add_messages.

# tool: just a value
@tool
def get_weather(city): return "72°F"
# ToolNode (simplified): wraps + writes to state
msg = ToolMessage(content=str(raw), tool_call_id=call_id)
return {"messages": [msg]}

So error dicts ({"error":…, "suggestion":…}) also become a ToolMessage the model reads next turn. ToolMessage has a status field ("success"/"error").

CategoryHandled whereTouches state?
Transient (rate limit, timeout, 503)retry with backoff below the model (RetryPolicy / tenacity)No — ideally invisible if retry succeeds
Permanent / bad-request (404, invalid args)surfaced to the LLM as an error ToolMessage so it self-correctsYes — this is the "error/suggestion dict" bucket
Programming (your bug)should fail loud, not be laundered into the conversationIdeally no
▲ handle_tool_errors trap

ToolNode's default handle_tool_errors=True catches everything — including your bugs — and surfaces them as error ToolMessages. Convenient, but it masks real programming errors. Many set it to False or pass a custom handler so bugs fail loud. If you want errors tracked separately for observability, add your own channel: errors: Annotated[list, operator.add] — a deliberate choice, not the default.

★ Interview line

Tools return plain values; ToolNode wraps them into ToolMessages in the messages channel — that's why tool output lands in state without the tool knowing about state. Transient errors retry beneath the model; bad-request errors become error ToolMessages the model self-corrects from; programming errors should fail loud, which means being careful with ToolNode's default error-swallowing.

Part II — Agent Memory

11Episodic memory — when it earns its place

The discriminator vs a log: a log is write-only (a human reads it offline); memory is read-back-at-inference (the agent reads it and acts differently). The test for every case: "Will the agent look this up later and change its action?" If no → it's a log with extra steps.

Cases that genuinely pass:

  • Failure avoidance — "last time in this env, sudo failed → use pip --user." The agent skips known dead ends. The strongest case.
  • Cross-session personalization — learns "this user wants type hints, prefers pytest" over weeks.
  • Self-mined few-shot — stores a successful trajectory, retrieves the closest one as a dynamic in-context example for a similar task.
  • Long-horizon resumption — reconstruct "what did I already try / conclude" after context is gone.
  • Loop-breaking — "I've tried 4 things in this family, all failed → escalate."

When not to: single-turn/short-session agents, when a semantic fact store suffices, when the lesson is universal (hardcode it), pure debugging (that's a log).

12Procedural vs episodic — the pipeline

Episodic = a record of a specific past event, kept as an instance, retrieved by similarity — answers "what happened?" Procedural = a generalized routine distilled from many events, kept as reusable know-how (prompt text, code, a skill file, fine-tuned weights) — answers "how do I do this?" Procedural is what you get when you compress many episodes into a skill and throw the instances away.

Procedural is the higher-leverage workhorse (cheap, reliable repeated work — Claude's Skills are procedural memory). But episodic is non-optional because: (1) it's the distillation source — no episodes, nothing to learn from; (2) irreducibly instance-specific recall (this user, this paused task); (3) concrete-example retrieval. Priority order: semantic facts > procedural routines > episodic.

★ Interview line

Procedural memory is distilled, generalized know-how — the workhorse for repeated work. Episodic is the record of specific instances, retrieved by similarity. The relationship is a pipeline: episodes are the raw material you distill into procedures. Episodic is non-optional because it's the distillation source and covers instance-specific recall that compresses into no general rule.

13Claude Dreaming — consolidation as a feature

Launched May 6, 2026 (research preview). A scheduled background process that reviews an agent's past sessions and rewrites its memory store — removing duplicates, replacing stale entries, surfacing new patterns. It does not touch model weights; it updates the external memory layer.

In our vocabulary: Dreaming is not a memory type — it's the consolidation operation between types. Input = episodic (session transcripts, up to ~100 per dream). Process = reflection/distillation. Output = procedural + semantic (preferences, recurring task patterns, learned facts, corrected beliefs), plus pruning of the stored set.

The name mirrors REM-sleep consolidation: the brain replays the day's events and turns hippocampal episodic traces into cortical semantic/procedural knowledge. It's exactly the episodic→procedural pipeline, run offline.

◆ Compression

Capture and consolidation are separate stages: experience capture is cheap and inline (sessions write episodes); experience consolidation is expensive and batched (the "dream" rewrites the durable store). That separation is the design lesson — and it's model-agnostic, so you can prototype the whole loop yourself.

▲ Source caveat

The concept and launch are confirmed; most specific benchmark numbers circulating come from secondary reporting, not Anthropic's own docs. Treat precise figures as "reported, not verified."

Part III — Routing & Tool Calling

14Routing: classifier vs cross-encoder

The category error to avoid: a cross-encoder is a classifier — a pairwise one. The real question is what you score against.

  • Classifier — scores query → label over a fixed closed set, one forward pass, jointly-trained decision boundaries.
  • Cross-encoder — scores (query, candidate) → relevance for arbitrary candidates (concatenated, full cross-attention), one forward pass per candidate. No fixed label set.
Closed, small, stable destinationsLarge / open / changing pool
UseClassifier (or LLM structured output)Cross-encoder rerank (on a cheap first-stage retriever)
Examplebilling / tech / sales routingtool retrieval — your Seller Copilot quote subagent
Cost1 forward pass total1 pass per candidate (so narrow first)

Your Seller Copilot use was correct: a large/changing tool catalog is a retrieval problem (retrieve-then-rerank), and a new tool is just an input, not a retrained class. The edge case: if routes grow large and dynamic (hundreds of specialists), routing becomes retrieval and the cross-encoder becomes right for routing too.

The real ladder in practice: LLM structured-output router (default, flexible, Literal[...]) → trained lightweight classifier (cheap/fast at volume on fixed routes) → cross-encoder rerank (destinations are a large/dynamic pool).

★ Interview line

A cross-encoder is a pairwise classifier scoring (query, candidate) for arbitrary candidates — it shines when destinations are large/open/changing, which is why tool selection is retrieve-then-rerank. Routing among a small fixed set is closed-set classification: one-forward-pass softmax with jointly-trained boundaries. Rule: closed-set → classifier, open/dynamic → cross-encoder; routing flips to the cross-encoder only once it has effectively become retrieval.

Part IV — Model Context Protocol (MCP)

16"Host" & the portability problem

"Host" here = the LLM provider (OpenAI, Anthropic, Google). Each built its own tool-calling format, and they didn't match — so the same tool needed three different shapes:

// OpenAI   {"type":"function","function":{"name":...,"parameters":{...}}}
// Anthropic {"name":...,"input_schema":{...}}
// Gemini    {"function_declarations":[{"name":...,"parameters":{...}}]}

And responses differ too (args as a JSON string vs a real object). So a tool was welded to one provider — swap models, redo the plumbing.

Two layers of fix: (1) Framework adapters — LangChain's bind_tools defines a tool once and translates to each provider's shape. Solves portability for your own tools in your codebase. (2) MCP — one standard protocol so independently-built tool servers plug into any agent on any provider, written once. The USB-C of tools.

17REST / gRPC / GraphQL / JSON-RPC — and why MCP

ProtocolOne defining idea
RESTact on noun-URLs with HTTP verbs (GET /users/42); stateless, one-shot. OpenAPI = its instruction manual.
gRPCcall remote functions like local; compact binary (Protocol Buffers); fast; internal microservices.
GraphQLclient requests exactly the fields it wants from one flexible endpoint.
JSON-RPCminimal "call this method with these params" as tiny JSON; transport-agnostic.

Two things plain web protocols didn't natively give for this job: (1) a standard runtime capability-discovery convention shaped for "LLM-callable tool" (REST/OpenAPI describes APIs to humans at build time, with no agreed runtime "enumerate your tools" endpoint); (2) bidirectional, stateful, streaming interaction (REST is stateless + one-directional + one-shot).

MCP picked JSON-RPC 2.0 over pluggable transports (stdio local, Streamable HTTP remote) — composing existing boring pieces rather than inventing new networking.

18JSON-RPC & transports — where the properties come from

The crucial correction: JSON-RPC by itself does NOT provide statefulness, bidirectionality, or streaming. It's just a message format. Those properties come from the transport — and JSON-RPC's one useful trait is being transport-agnostic, so you pick a transport that has them.

  • Bidirectionality — JSON-RPC has notifications (no id, no response expected), and either side can send them. Over a persistent two-way pipe, the server can push "resource changed" unprompted.
  • Statefulness — the transport holds one connection open across many messages; the first exchange is a capability-negotiation handshake; everything after lives in that remembered session.
  • Streaming — entirely the transport's job: a stdio pipe you keep writing to, or SSE (Server-Sent Events) holding the HTTP response open to push event after event.

Why JSON-RPC at all, then? You still need a uniform message format both sides agree on, regardless of transport. JSON-RPC is the smallest such format, transport-agnostic, with the request/notification split bidirectionality needs.

◆ Compression

JSON-RPC = tiny uniform message format with a request-vs-notification distinction. Statefulness / bidirectionality / streaming come from the transport (persistent stdio pipe, or HTTP+SSE where SSE streams server→client). "Pluggable transports" = the same messages ride whichever pipe fits the deployment.

19Why not REST — the plain-words pitch

REST is built for a human to read docs, write code against fixed endpoints, ship. Stateless, one-directional. Wrong for agents three ways:

  1. An agent needs to connect to an unseen server and ask at runtime "what can you do?" and get a machine-readable tool list. REST has no standard for that → custom glue per server.
  2. The server sometimes needs to talk back on its own ("data changed") or stream a long tool's progress. Plain REST can't do server-initiated or streaming.
  3. The real point: even if you bolted all that on, everyone would do it differently → back to per-pairing integration. The problem isn't capability, it's the lack of a shared standard. Like USB-C — not electrically superior, just universally agreed, so any tool plugs into any agent. Turns N×M into N+M.
★ One-breath version

REST is stateless, one-directional, and has no standard runtime tool-discovery — agents need stateful, two-way, streaming connections plus auto-discovery, but mostly they need everyone using one standard so any agent works with any tool. MCP is that standard; the USB-C of tools.

20tools/list — runtime discovery

Every MCP server answers one standard question the same way:

Agent → server:  {"method":"tools/list"}        # "what can you do?"
Server → agent:  [{"name":"create_issue",
                   "description":"Open a new issue",
                   "input_schema":{"repo":"string","title":"string"}}, …]

The agent now knows, at runtime, what it can call and with what args — no GitHub-specific code. Swap to a Slack server tomorrow: same tools/list call, Slack's tools back, zero new glue. With REST you'd read each API's docs and hand-code an integration per service. tools/list is exactly what REST never standardized.

21Change notifications — narrow but real

notifications/tools/list_changed only matters during a persistent open session. Example: an IDE agent (Claude Code, Cursor) you keep open for hours; mid-session you enable a new integration; the server pings list_changed; the agent re-runs tools/list and can use the new tool without a restart.

For typical request-response agents (fire query → run → finish → close), there's no one listening, so it rarely fires. It's a feature for long-lived sessions, not one-shot calls — which is why it felt obscure.

22Context bloat — MCP vs CLI (2026)

Why MCP bloats context: the protocol declares every tool's full schema up front (via tools/list) before the agent reads your request — and that payload is often re-sent every turn. Reported: tool definitions burning 55,000+ tokens before a single user message; Perplexity's CTO cited up to 72% of the window consumed pre-input. It also degrades reasoning by crowding the high-attention part of the window.

Why CLI wins: (1) near-zero upfront context — an ~80-token prompt + progressive disclosure via --help; (2) the model already knows common CLIs (git, grep, jq, curl, aws, psql) from training, so it needs no schema at all and can one-shot them.

▲ Don't overcorrect to "MCP is dead"

CLI has its own bloat — on the output side (raw git log can dump 10k unstructured lines; MCP returns bounded, structured output). The CLI edge narrows on complex chained writes. And MCP's overhead buys governance for multi-tenant, compliance-sensitive, customer-facing systems. Switching transport doesn't fix the threat model (prompt injection survives). 2026 consensus: CLI for dev workflows, MCP where typed structure & governance matter.

★ Interview insight

The real axis is eager vs lazy tool exposure: MCP declares everything upfront (pay in tokens), CLI discovers on demand. Second-order point most candidates miss: CLIs exploit the model's pretraining prior on Unix tools — no schema needed because the composability grammar is in the weights.

23What a CLI actually is, mechanically

A CLI is just an installed program you run by typing a command; it takes text args, prints text. The trick: the agent uses ALL CLIs through ONE generic bash/shell tool — one tool, not N typed tools.

Agent emits:  bash(command="slack-cli send --channel general --text 'hi'")
Runtime runs that string in a real shell → stdout/stderr text back to agent

How it knows what to type, without schemas: (1) the model already knows the command from training; (2) --help — it runs help as a bash call to learn syntax lazily, only for the tool it's using, only when needed; (3) an ~80-token system-prompt hint.

"How do you inject it?" — two steps, no per-tool registration: install the binary on the box (so it's on PATH) + give the agent a shell tool. Claude Code has a bash tool built in, which is exactly why "Claude Code + CLIs" is the popular combo — every installed CLI is instantly usable with zero registration.

And yes — a Slack CLI can exist. The choice was never "does a CLI exist" (one usually can). It's: model already knows it / you just want cheap access → CLI; you need typed structure, auth scoping, bounded output, audit, cross-agent reuse → MCP.

◆ Compression

With raw CLI the LLM-generated string is the security boundary — dangerous in multi-customer systems. MCP moves the boundary into trusted code: typed args cage requests, the server (not the model) attaches credentials and checks permissions, output is bounded, every call is auditable. That control is what MCP's overhead buys — and it only matters when the agent acts on behalf of multiple parties with real blast radius. On your own dev box, none of it applies → CLI wins.

25Building & hosting an MCP server

Why build one: so AI agents can do things in your product (press the buttons humans click). Payoffs: distribution (be the tool the agent reaches for, across all clients), write-once over any provider, usually a thin shim over an API you already have, and governed internal automation. Skip it if your app has no actions worth automating.

from mcp.server.fastmcp import FastMCP
mcp = FastMCP("my-app")
@mcp.tool()
def get_order_status(order_id: str) -> str:
    """Look up the status of a customer order."""  # → tool description the LLM reads
    return db.lookup(order_id)
if __name__ == "__main__":
    mcp.run()   # stdio by default

The SDK generates the schema from your type hints (so hints/Pydantic become the contract), turns the docstring into the description, and handles tools/list + JSON-RPC + transport.

Where/how to host:

  • Local (stdio) — not really hosted; the client launches it as a subprocess and talks over pipes. Just point the client config at the run command. Personal/dev tools.
  • Remote (Streamable HTTP) — a web service. Deploy to Cloud Run / AWS / Fly.io / a container, give it a URL, put OAuth 2.1 + PKCE in front, clients connect over HTTP+SSE. For multi-user/customer-facing.

26Resources vs tools (vs prompts)

MCP servers expose three primitives: tools, resources, prompts.

  • Tool = a verb. "Do something" — runs, may have side effects, one-off result. Model-controlled (the LLM decides to call it). Send email, run query, create ticket.
  • Resource = a noun. Addressable readable data by URI (weather://Delhi/current), fairly stable, re-read across turns, subscribable to changes. App/user-controlled (like @-mentioning a file). Config, KB docs, current state.
  • Prompt = a pre-written template the user can trigger. Nice-to-have; build last.

Designing a server: enumerate the actions → tools (≈80% of what you build); enumerate frequently-read stable data → resources (optional, patchy client support, often just done as a read-tool). When in doubt, ship a tool.

◆ Compression

Tool = verb, action, one-off result, model decides, side effects. Resource = noun, addressable readable data, app attaches it, stable, re-read, subscribable. "Do this" → tool; "here's reference data to keep available" → resource.

27OAuth 2.1 + PKCE (remote MCP auth)

  • OAuth = log in without giving the app your password; the provider hands the app a scoped token instead.
  • OAuth 2.1 = the modern hardened version (insecure options removed, best practices mandatory).
  • PKCE ("pixie") = anti-theft for the auth code: the app generates a secret, sends only its hash, and proves it holds the original when redeeming the code → an intercepted code alone is useless.
  • Resource Indicators (RFC 8707) = pin each token to one specific service, so a rogue server can't replay a token meant for it against a different service.
◆ Compression

OAuth = scoped token instead of your password; 2.1 = hardened version; PKCE = makes an intercepted auth code worthless without a secret only the real app holds. MCP uses it so remote servers get safe, scoped, password-free access on your behalf — Resource Indicators bind each token to one service so a rogue server can't reuse it elsewhere.

Part V — Agent-to-Agent (A2A)

28The A2A protocol — increasing, vendor-neutral

Adoption is rising, not declining, and it's not tied to Google ADK anymore. At its one-year mark (April 2026): 150+ supporting organizations; native integration in Azure AI Foundry, Amazon Bedrock AgentCore, and Google Cloud; production deployments in supply chain, finance, insurance, IT ops. Search interest grew ~22% MoM. Strongest signal it's winning: IBM's competing ACP merged into A2A (Aug 2025) — effectively no real alternative as of 2026.

The "needs ADK" assumption is outdated: Google donated the protocol + spec + SDKs to the Linux Foundation in June 2025, making it vendor-neutral (no more ADK-bound than Kubernetes is Google-bound). v1.0 (early 2026) made it production-grade: Signed Agent Cards (cryptographic trust for discovery, blocks card-forgery), multi-tenancy, multi-protocol bindings (JSON-RPC + gRPC), version negotiation.

★ Interview line

A2A is increasing, not declining — 150+ orgs, native in Azure/Bedrock/Google Cloud, competitor ACP absorbed into it. It's vendor-neutral (Linux Foundation since June 2025), so it's not ADK-bound. Mental model: MCP = vertical (agent↔tool), A2A = horizontal (agent↔agent) — complementary layers, not rivals. Discovery is via Agent Cards, cryptographically signed since v1.0.

▲ Honest scope caveat

The traction is mostly big-company / cloud-platform / enterprise inter-agent plumbing. As a solo dev not building cross-vendor multi-agent systems, you may never touch A2A directly — which is why it felt niche. Big in enterprise, quieter in solo-dev land. Both true at once.
