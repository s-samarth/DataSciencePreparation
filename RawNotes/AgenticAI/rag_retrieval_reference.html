SECTION 01 — FOUNDATIONSChunking strategies

How you split documents before embedding decides the ceiling on retrieval quality. No reranker fixes a bad chunk.

Recursive chunking

Split text by a priority list of separators, biggest semantic unit first, dropping finer only when a chunk is still too big. Order: \n\n (paragraphs) → \n (lines) → . (sentences) → (words) → characters.

Logic: try splitting on \n\n; if any piece still exceeds chunk_size, re-split that piece on \n, then . , and so on. You cut at the most natural boundary that fits the budget instead of slicing blindly at character N. This is the 2026 default for unstructured prose — preserves meaning better than fixed-size, without semantic chunking's embedding cost.

Parent-child chunking

Decouple what you match against from what you feed the LLM. Index small chunks for retrieval precision, return their larger parent for generation context.

  1. Split doc into large parent chunks (~2048 tok).
  2. Split each parent into small child chunks (~256 tok).
  3. Embed and index only the children.
  4. At query time, match a child → but return its parent.
Why it worksSmall chunks embed cleanly (one idea = sharp match) but a stranded 256-token hit often lacks context to answer well. So match small, serve big. The cost: more tokens per query, and over-fetch if parents are huge.

Chunk sizing in practice

  • General range: 256–512 tokens for the embedded (child) chunk. Smaller = sharper embeddings; larger = more context but blurrier vector.
  • Dense / conditional text (legal, policy) pushes toward the larger end (512) or parent-child, because a clause's meaning depends on definitions stated nearby — chopping too small severs the logic.
  • Overlap ~10–15% catches rules that straddle a boundary.
  • Count tokens with the same tokenizer the embedder uses — otherwise a "512-token chunk" may truncate silently.
Honest positionChunk size is tuned, not decreed. The right answer is to sweep sizes against your golden eval set and measure nDCG@k — the optimum depends on your query and document distribution. A confident single number is a weak answer; "here's my prior, and I'd measure it" is the strong one.

SECTION 02Why not just a CSV? Where vectors live

Retrieval isn't a lookup, it's a similarity search: given a query vector, find the top-k closest of N. A CSV + Python can only brute-force — compute similarity against every vector, every query: O(N·d).

  • ~1,000 chunks → numpy is fine, milliseconds. Genuinely no DB needed.
  • ~100K → sluggish.
  • 10M+ → dead on arrival.

What a vector DB actually adds is an ANN index (HNSW, IVF) — approximate nearest neighbor, trading a sliver of recall to go from O(N) to ~O(log N). That's the whole game.

What a flat CSV lacks

  • No concurrency — one process writing at a time; two simultaneous writers corrupt the file. A DB handles many readers/writers with locking.
  • No metadata filtering — you often want "similar chunks where source='invoices' AND date>2024". A DB filters during search; CSV forces a manual Python loop, unindexed.
  • No persistence guarantees — a crash mid-write leaves a half-broken file. DBs use transactions + write-ahead logs so data survives crashes.

The decision ladder

TierExamplesUse when

On-disk embeddedChroma, LanceDB, DuckDB-VSSExceeds RAM but you want zero ops — index spills to disk, no server.

The decision ruleYou graduate tiers when you hit a specific wall: RAM (→ on-disk), concurrency/writes (→ server), or ops/scale (→ SaaS). Most systems that "need Pinecone" run great on pgvector or LanceDB. The interview question is never "which DB" — it's "what's your N, QPS, write pattern, and latency budget?" Answer those four and the DB picks itself.

SECTION 03Embedded vs. server — deployment models

"Embedded," "SQLite-like," "Postgres extension" describe how a DB runs, not what it stores.

ModelMeaningVector examples
EmbeddedRuns inside your app's process as a library. No server, no network hop. pip install and go.Chroma, LanceDB, FAISS
SQLite-likeEmbedded + stores to a single file on disk, zero server. Borrows SQLite's reputation as the gold standard of simple file-based DBs.LanceDB, Chroma
Postgres extensionA plugin (pgvector) adding vector search into existing Postgres. Vectors sit in a normal table beside relational data, queried with SQL.pgvector

Key clarifications

SQL vs. Postgres: SQL is a language; PostgreSQL is a product that speaks it (so do MySQL, SQLite, SQL Server). "Postgres vs SQL" is a category error — Postgres is a SQL database.

SQLite is a famous library storing an entire SQL database in one file, with no server — the engine is compiled into your app. Postgres/MySQL run a separate server you connect to over a socket; SQLite reads/writes a single .db file from inside your process.

Why most DBs run as servers

Not for "decoupling code changes" — that's a minor side benefit. The real reasons are runtime properties an embedded library can't give:

  • Shared state across many clients — API, batch jobs, dashboards, teammates all hit the same live data. Embedded lives in one process's memory.
  • Concurrent writes — servers handle thousands of simultaneous reads+writes with transaction isolation.
  • Data outlives any process — app can crash/restart/scale; the server and data are untouched. Run many stateless app copies against one DB.
  • Resource isolation — give the DB a beefy machine, scale the app layer separately.
Chroma vs Qdrant for a DS teamNotebook / experiment / one person → Chroma (embedded). Shared index the team queries / production / an API → Qdrant (server). The trigger to go server is never "we felt like it" — it's a concrete event: a second consumer appears, you need writes while serving, or the index outgrows one machine's RAM.

SECTION 04 — ANN ALGORITHMSHNSW — Hierarchical Navigable Small World

The 2026 default in-memory ANN index. Built from two ideas stacked together.

Idea 1 — Small-world graphs, navigate by greedy hops

Like "six degrees of separation": most links are local, a few are long-range, and those long links let you cross the network fast. Each vector is a node linked to nearby vectors plus a few longer links. To search, start somewhere and greedily walk toward the query — move to whichever neighbor is closest to the query, repeat, stop when no neighbor is closer. The flaw: a flat graph gets stuck in tiny local steps early on. You need big jumps first, small jumps near the end.

Idea 2 — Hierarchy (the skip-list trick)

Layer 0 has every node; each layer up keeps ~1/M of the layer below (exponentially fewer). Top = sparse with long links; bottom = dense. Search descends top-down: greedily walk each layer toward the query, drop down a layer, repeat; do the thorough search on layer 0. Top layers are an express highway to the right city; layer 0 is local streets to the exact address.

Library usage & the three hyperparameters

import hnswlib
index = hnswlib.Index(space='cosine', dim=768)
index.init_index(max_elements=N, M=16, ef_construction=200)
index.add_items(data, ids)
index.set_ef(50)        # ef_search — tune live, no rebuild
labels, dist = index.knn_query(query, k=10)
ParamWhat it doesTrade

Strengths, weaknesses, the ceiling

  • Strengths: SOTA recall/latency in RAM; ~O(log N) search; query-time tunable; no training step (unlike IVF).
  • Weaknesses: memory hungry — graph links cost RAM on top of vectors (the defining limit); slow/expensive builds & inserts (each insert runs a search); deletes are awkward (tombstone + periodic rebuild); all in-memory (DiskANN for on-disk).
Interview trap"How does HNSW scale to a billion vectors?" — honest answer: it doesn't, gracefully. That's where IVF-PQ, DiskANN, or sharding come in. Knowing the RAM ceiling is the senior signal.

SECTION 05IVF — Inverted File index

Name borrowed from text search: an inverted file maps word → docs containing it. IVF does the spatial version — partition vector space into regions, map each region → the vectors in it. At query time, find which region(s) the query lands in and scan only those.

Yes — it's k-means

You partition by running k-means with nlist clusters. Each centroid defines a Voronoi cell. The centroids are the inverted file's keys.

How k-means picks/moves centroids

  • Init (k-means++): first centroid random, then each next chosen with probability ∝ squared distance from the nearest existing centroid — so points far from existing centroids are likelier picked. This spreads them out (the "diverge" property), avoiding clumping.
  • Lloyd's loop: (1) assign every point to nearest centroid; (2) move each centroid to the mean of its points; repeat until stable.

Centroids aren't engineered equidistant — they settle into the data's density; k-means++ just gives a good spread-out start.

Parameters

  • nlist — number of clusters. Rule of thumb ≈ √N. More cells = less to scan each, but more centroids to compare.
  • nprobe — cells scanned per query. The recall/speed dial (IVF's analogue of ef). Tunable live. nprobe=1 fastest but misses border neighbors.

They're different points on the memory-vs-quality frontier — which is why FAISS combines them.

SECTION 06Product Quantization (PQ)

Problem: vectors are big. A 768-dim float32 = 3072 bytes; a billion = 3 TB, won't fit in RAM. PQ compresses each vector into a tiny code (e.g. 8 bytes).

How PQ encodes — the "product" trick

  1. Split each vector into m sub-vectors (768-dim, m=8 → 8 chunks of 96 dims).
  2. Quantize each chunk with its own k-means (256 centroids per subspace, fits in 1 byte).
  3. Vector → m bytes (the centroid ID per subspace). 3072 bytes → 8 bytes, ~380× compression.

"Product" = the full approximation is the Cartesian product of per-subspace codebooks: m=8 subspaces × 256 options = 256⁸ ≈ 18 quintillion representable vectors, from only 8×256 stored centroids. Combinatorial power, tiny storage.

Search without decompressing — ADC

Asymmetric Distance Computation: the query stays full-precision; database vectors are quantized (hence "asymmetric").

  1. Split the query into the same m sub-vectors.
  2. Build a lookup table: for each subspace, distance from the query's sub-vector to all 256 centroids → an 8×256 table. Cost: 2048 small distances, once per query.
  3. Score each database vector by its code: table[0][42] + table[1][7] + ... — m lookups + m adds. No per-vector distance math.

Squared L2 is separable across dimension-chunks, so total distance = sum of per-subspace distances = sum of m table lookups. One vector: 768 multiply-adds → 8 lookups+adds (~50–100× faster) plus ~380× smaller.

Why PQ pairs with IVF (not HNSW)

  • IVF + PQ are complementary: IVF answers "which vectors to look at" (prune by cell); PQ answers "how to store/compare each cheaply" (compress). Alone, IVF still stores fat vectors; alone, PQ must scan all N codes. Together (IVFPQ) = prune to a few cells and every vector is 8 bytes → billion-scale in RAM.
  • IVFPQ stores the PQ code of the residual (vector minus cell centroid) — residuals are smaller/more similar within a cell, so PQ quantizes them more accurately. IVF gives PQ a better-conditioned input.
  • Why not PQ on HNSW: HNSW's greedy graph navigation needs accurate distances to pick the next hop; PQ's approximate distances make worse hops → recall degrades more than for IVF (where PQ only ranks an already-pruned shortlist). Also HNSW's real memory hog is the links, which PQ doesn't shrink. (HNSWPQ exists but the value is weaker.)
  • Why not PQ alone: works ("flat PQ") but scans all N codes — fine to a few million, want IVF's pruning beyond.
What PQ costsRecall — it's lossy. Standard fix: re-ranking — IVFPQ gets a coarse top-100 fast, then re-score those against the original full-precision vectors for an accurate top-10. Same coarse-then-exact philosophy as parent-child.
import faiss
quantizer = faiss.IndexFlatL2(d)
index = faiss.IndexIVFPQ(quantizer, d, nlist, m, nbits)
index.train(train_vec)   # learns k-means centroids AND PQ codebooks
index.add(db_vec)        # assigns to cells, stores PQ codes of residuals
index.nprobe = 16
D, I = index.search(query, k=10)

SECTION 07LSH & KD-trees — the casualties of high dimensions

LSH — Locality-Sensitive Hashing

Normal hashing scatters similar inputs apart; LSH does the opposite — hash functions where similar vectors collide on purpose. Hash the query, look only in its bucket.

Random hyperplane hashing (for cosine): a random hyperplane gives each vector 1 bit (which side). b hyperplanes → a b-bit signature; matching signatures = same bucket. Two vectors agree on a random plane with probability ∝ their angular closeness. Use L independent tables; a vector is a candidate if it collides in any table.

  • Dials: b (bits/table — stricter buckets) and L (tables — more recall, more memory).
  • Strengths: O(1) hashing, simple, distributable, trivial streaming inserts, strong theoretical guarantees.
  • Weakness: needs many tables for HNSW-level recall → Pareto-dominated for dense embeddings.
  • Still alive for: extreme-scale distributed/streaming; and MinHash for Jaccard on sets (dedup, near-duplicate detection — used to dedupe LLM training corpora).

KD-tree — k-dimensional tree

Binary search generalized to space: recursively median-split, cycling axes (x, then y, then z…). Search descends to the query's leaf, then backtracks, pruning branches whose box is too far to contain anything closer.

  • Strength: exact NN and genuinely fast in low dimensions (≤ ~10–20) — geo, point clouds, small tabular k-NN.
  • Fatal weakness — curse of dimensionality: as dims grow, almost everything is roughly equidistant, the splitting planes stop separating near from far, pruning fails, and search degrades to ~O(N) (worse than brute force with overhead). At 768/1536 dims, useless.
The unifying threadKD-trees fail because exact spatial-partitioning pruning dies in high dimensions. LSH and the graph/cluster methods (HNSW, IVF) survive precisely because they're approximate and don't rely on clean geometric partitioning. "Why not a KD-tree for embeddings?" → curse of dimensionality; pruning becomes ineffective so it degenerates to a linear scan.

SECTION 09Matryoshka embeddings & dimension truncation

text-embedding-3-small (1536-dim) and -large (3072-dim) support Matryoshka-style truncation via the dimensions parameter — you can shorten a vector (e.g. 1536→512) with only minor quality loss. Matryoshka training packs the most important information into the leading dimensions, so prefixes remain useful embeddings on their own. Lets you trade a little accuracy for big memory/speed savings without re-embedding.

SECTION 10 — RETRIEVAL MECHANICSHybrid retrieval & fusion

Combine BM25 (lexical) and vector (semantic) first-stage retrieval. The 2026 default for text because each catches what the other misses:

  • BM25 nails exact tokens vector search mangles: acronyms, SKUs, error codes, dollar thresholds, clause names.
  • Vector catches paraphrased questions where wording doesn't match the document.

Fuse the two ranked lists with Reciprocal Rank Fusion (RRF) — parameter-free, robust, combines by rank position without needing to tune a weight between two incomparable score scales. The production-default fusion method.

SECTION 11TF-IDF & BM25

TF-IDF

Score per term per document = TF × IDF, summed over query terms.

  • TF — term frequency in this doc (how much it's about the term).
  • IDF — rarity across all docs. The log lives here:
IDF(t) = log( N / df(t) )

N = total docs, df(t) = docs containing t. Why the log: without it, IDF grows linearly and over-rewards ultra-rare terms; the log compresses that. A term in every doc → N/df=1 → log(1)=0 → contributes nothing. High TF-IDF = frequent here, rare elsewhere = distinctive.

BM25 — TF-IDF with two fixes

Same skeleton (TF-part × IDF-part, summed), with two corrections. The "weird terms" are the knobs implementing them.

BM25(t,d) = IDF(t) · f · (k1 + 1)f + k1 · (1 − b + b · |d|/avgdl)
  • Fix 1 — TF saturates (via k1, ~1.2–2.0): repeated terms hit diminishing returns instead of growing linearly. Spamming a keyword stops helping. The biggest behavioral difference.
  • Fix 2 — length normalization (via b, ~0.75): documents longer than avgdl get discounted so they don't win just by being long. b=0 ignores length, b=1 full normalization.
Interview line"BM25 is TF-IDF with two corrections: term frequency saturates so spamming a keyword stops helping, and it normalizes for document length so long docs don't win by length alone. k1 tunes saturation, b tunes the length penalty. That's why BM25 is the production default — the fixes match how relevance actually behaves."

SECTION 12Elasticsearch & Apache Lucene

Elasticsearch is fundamentally a BM25 lexical/full-text search engine (and distributed document store) — not natively a hybrid. It dominated full-text search for over a decade (site search, e-commerce, the ELK stack for logs). Modern versions (8.x+) added dense vector + ANN (HNSW), so you can now do BM25+vector hybrid in it — but its DNA is keyword search, vectors bolted on later.

  • Shows up in JDs because it's everywhere as infrastructure (logging, search, analytics, now RAG backend). "Elasticsearch experience" often means "we run a big search/logging system."
  • vs. dedicated vector DBs: native vector DBs can be leaner for pure-vector workloads; Elasticsearch's pitch is mature lexical + vector + filtering in one scalable system you may already run.
  • OpenSearch = the open-source fork after a license change; often interchangeable in JDs.

Apache Lucene

The foundational Java search library — the actual engine that builds the inverted index, runs BM25, and (now) stores vectors + HNSW. Not run alone; products are built on it. Lucene is the engine; Elasticsearch / OpenSearch / Solr are the distributed cars built around it. When people say Elasticsearch "is BM25 under the hood," that implementation literally is Lucene.

SECTION 13Cross-encoder reranking

Example model: cross-encoder/ms-marco-MiniLM-L-6-v2 — the standard reranker baseline.

Architecture

A distilled BERT (MiniLM): 6 transformer layers ("L-6"), hidden size 384, ~22M params — small/fast by design. The defining cross-encoder property is the input:

[CLS] query tokens [SEP] document tokens [SEP]
   → 6 MiniLM layers (joint self-attention)
   → take [CLS] final hidden state
   → linear head → single relevance score

Query and doc are concatenated into one sequence, so self-attention runs across both jointly at every layer — the model sees them interacting. That's why cross-encoders are far more accurate than bi-encoders (which encode separately, compare only at the end).

Bi-encoder (retrieval)Cross-encoder (rerank)
Inputquery & doc separatelyconcatenated, together
Outputtwo vectors → cosineone relevance scalar
Interactionnone until the endfull attention every layer
Speed / usefast, retrieve over millionsslow, rerank a shortlist

Can't precompute (needs both inputs together) → can't run over the whole corpus → used as a reranker on ~20–50 candidates: retrieve cheap with bi-encoder, rerank precise with cross-encoder.

Training & fine-tuning

  • Base: MiniLM pretrained by distillation (student mimics a larger BERT/RoBERTa).
  • Reranker fine-tune: on MS MARCO (real Bing queries + relevant/irrelevant passages) → the scalar learns "relevance."
  • Adapting to your domain: continue-train the ms-marco checkpoint on domain triplets (query, positive, negative) — crucially with hard negatives (topically similar but irrelevant). Random negatives teach little.
When to fine-tune vs use off-the-shelfOff-the-shelf ms-marco generalizes well — a fully defensible default. Fine-tune when the general model can't tell your relevant chunks from your hard negatives — i.e. when domain-specific relevance distinctions are too subtle (e.g. a current rule vs a superseded near-identical one). The trigger is confusion about what "relevant" means in your domain, not ranking precision.
Reranker as filter vs rankerUsing it to get the right chunks into a tight top-5–8 is set selection (precision filter), subtly different from fine-grained ordering. Within that small set, order matters less (the LLM reads all of them) but not zero — position bias / "lost in the middle" means the single most decisive chunk still belongs near the top. Say "ranking mattered less than set-membership for my config," never "ranking didn't matter."

SECTION 14InfoNCE

Information Noise-Contrastive Estimation. The CE traces to Noise-Contrastive Estimation, and in practice it's a cross-entropy over similarity scores — pulling a positive pair together against a set of negatives. The standard contrastive loss for training embedding models.

SECTION 15 — RAG ARCHITECTURESBeyond-naive patterns: HyDE → GraphRAG

Each pattern fixes a different failure mode of naive RAG. Frame each as "naive RAG fails when X → the pattern addressing X is Y."

HyDE — Hypothetical Document Embeddings

Fixes: query≠answer embedding gap (questions are phrased unlike the text answering them). How: LLM hallucinates a fake answer; embed that (not the query) for retrieval. The fake answer, even if wrong, sits near real answers in embedding space — matching answer-to-answer. Cost: one extra LLM call; fails if the model can't even fake a plausible answer.

RAPTOR — Recursive Abstractive Processing for Tree-Organized Retrieval

Fixes: chunks lose the big picture. How: at index time, cluster chunks → summarize each cluster → cluster the summaries → summarize again, recursively → a tree. Index every level; detail questions hit leaf chunks, thematic questions hit summary nodes. Cost: expensive to build (lots of summarization); summaries can drift. For corpora with both granular and synthesis questions.

Self-RAG — Self-Reflective RAG

Fixes: always retrieves & always trusts results. How: model emits reflection tokens to (1) decide whether to retrieve at all, (2) judge each passage's relevance, (3) check whether its output is actually supported by evidence (self-grounding). Can re-retrieve or discard. Cost: more calls; the "is this supported?" step is the gold.

CRAG — Corrective RAG

Fixes: the bad-retrieval case specifically. How: a retrieval evaluator scores the set Correct / Ambiguous / Incorrect, then branches — Correct → refine & use; Incorrect → fall back to web search; Ambiguous → both. Distinction from Self-RAG: Self-RAG reflects on its own generation; CRAG corrects the retrieval with an external safety net.

GraphRAG

Fixes: can't connect facts across documents (multi-hop, global "connect-the-dots"). How: at index time an LLM extracts entities + relationships → a knowledge graph; clusters into communities with pre-made summaries. Query traverses the graph / uses community summaries. Cost: expensive to build, quality depends on extraction. For investigations, compliance, richly interconnected KBs. Overkill for simple lookup.

SECTION 16Agentic RAG

Core intuition: everything above is a fixed pipeline — the path is designed at build time (even Self-RAG's reflection is a scripted checkpoint). Agentic RAG replaces the fixed pipeline with an LLM that decides what to do at each step in a loop. The control flow is generated at runtime by the model. Traditional RAG follows a path you designed; agentic RAG decides the path as it goes.

What the loop unlocks

  • Multi-step / iterative retrieval — retrieve, see what's missing, retrieve again with a refined query (the big one; impossible single-shot).
  • Query decomposition — break a compound question into sub-questions, retrieve each, synthesize.
  • Tool/source routing — choose vector DB vs SQL vs web vs calculator per sub-task.
  • Self-correction — change strategy on failure, not just flag it.

Tradeoffs — it is NOT strictly better

Fixed pipelineAgentic
Control flowfixed, you design itLLM decides at runtime
Latency / costlow, predictablehigh, variable (N loops)
Debuggabilityeasy (fixed path)hard (non-deterministic)
Failure modebad answerloops, wrong tool, runaway cost
Best forsimple single-hopcomplex multi-hop / multi-source
Decision rule + the trapMatch architecture to query complexity. Simple/homogeneous/latency-sensitive → pipeline. Complex/multi-source/quality-over-speed → agentic. Mature systems use a router sending easy queries to a cheap pipeline and hard ones to the agent. Trap: "if it's more capable, why not use it for everything?" → cost, latency, debuggability, and capability is wasted on simple queries. The earlier patterns become tools the agent can choose to invoke; CRAG's web-fallback is one hardcoded agentic decision that agentic RAG generalizes.

SECTION 17 — EVALUATIONMRR & nDCG@k

Both measure "did the good stuff land near the top," with different assumptions.

MRR — Mean Reciprocal Rank

Assumes one right answer. Find the rank of the first relevant result, take its reciprocal, average over queries.

MRR = (1/Q) · Σ ( 1 / rank_of_first_relevant )

Rank 1 → 1.0, rank 2 → 0.5, rank 3 → 0.33, not found → 0. Ignores everything after the first hit. Use for binary/singular relevance (factoid QA).

nDCG — Normalized Discounted Cumulative Gain

Assumes graded relevance and that all top-k positions matter with diminishing weight. The primary metric for reranking.

DCG@k = Σi=1..k ( reli / log₂(i + 1) )
nDCG@k = DCG@k / IDCG@k
  1. Gain = each result's graded relevance (e.g. 0–3).
  2. Discount = divide by log₂(rank+1): rank 1 divisor = log₂(2)=1 (full credit), rank 2 = log₂(3)≈1.585 (discounted)…
  3. IDCG = DCG of the perfect ordering; normalize → 0 to 1, comparable across queries.
One-line splitMRR = "how high is the first good one" (binary, single answer). nDCG = "how good is the whole ordering" (graded, all positions). For reranking, lead with nDCG; graded labels are what make it meaningful (with binary labels it collapses toward simpler metrics).
The real hard partThe metrics are trivial arithmetic. The engineering is getting the relevance labels — you author the relevance definition (e.g. a 0–3 rubric with examples). Lead an interview answer with "first I'd build a labeled eval set, here's how," not the nDCG formula.

SECTION 18Contextual precision & recall

RAGAS-style metrics judging the retrieved set; used here via the DeepEval library (RAGAS definitions). LLM-judged — the judge decides relevance against the expected answer.

Contextual recall — "did I get everything?"

Of the information the correct answer needs, how much is in the retrieved chunks. Computed by breaking the ground-truth answer into claims and checking the fraction supported by retrieved context.

recall = (reference-answer claims supported by context) / (total claims)

Contextual precision — "is the good stuff ranked high?"

Whether relevant chunks appear above irrelevant ones. Rank-aware by design (mechanically like Average Precision / MAP).

Worked precision example

Retrieved 5 chunks, relevance: rank1=1, rank2=0, rank3=1, rank4=1, rank5=0. Precision@k only at ranks holding a relevant chunk:

  • rank 1: 1/1 = 1.00 · rank 3: 2/3 = 0.67 · rank 4: 3/4 = 0.75
  • Average over the 3 relevant positions: (1.00+0.67+0.75)/3 = 0.81

Each relevant chunk's contribution = "how clean is the list up to and including me." Relevant-near-top → high precision@k; relevant-buried-below-junk → low. Same chunks in worse order → lower score.

Why rank-aware? And do we always need it?

Built rank-aware because retrieval feeds a position-sensitive LLM — the "lost in the middle" effect means a buried relevant chunk underperforms a top one. The metric predicts generation quality, not just retrieval correctness.

2026 nuanceRank-awareness is a hedge against LLM position-sensitivity; its importance scales with (a) how position-sensitive the model is and (b) how large the retrieved set is. For a small top-5–8 fed to a strong long-context model after a reranker, order matters less — a precision-as-filter framing is defensible. It scales back up the moment you widen k or use a more position-sensitive model. Say "ranking mattered less for my config," not "ranking doesn't matter."

The recall critique (and why it's valid)

RAGAS recall measures against the reference answer's claims, NOT the complete relevant-document set in your DB. So it conflates two different failures:

  1. Relevant chunk exists in DB but retrieval missed it → true retrieval failure (should be penalized). ✓
  2. Information was never in the corpus → still penalized, though retrieval did all it could. ✗ — a corpus-coverage gap blamed on retrieval.
The defensible framing"RAGAS recall is measured against the reference answer, so a corpus gap is penalized like a retrieval miss. True retriever recall needs the complete relevant-set labels, infeasible at 10–20K docs. So rather than a recall metric whose semantics didn't match what I wanted, I used contextual precision and graded nDCG — which I could compute rigorously." Don't say "recall is impractical"; say "this operationalization conflated corpus coverage with retrieval, and true recall needs labels I couldn't build."

SECTION 19Faithfulness

A generation metric (nDCG/precision were retrieval metrics): is every claim in the answer supported by the retrieved context? Catches within-RAG hallucination — retrieval was fine but the model invented beyond the source.

Critical: faithfulness does NOT measure correctness — only grounding-in-context. An answer can be faithful-but-wrong (context was wrong) or correct-but-unfaithful (used parametric knowledge the context lacked).

How it's computed (LLM-judge, two steps)

  1. Claim decomposition — judge breaks the answer into atomic claims.
  2. Verification — for each claim, can it be inferred from the retrieved context? yes/no.
faithfulness = (claims supported by context) / (total claims)

It's the rigorous version of a cosine-similarity grounding proxy — claim-level entailment catches paraphrased support that similarity misses.

Golden set — cheaper than for retrieval

Reference-free: measured against the retrieved context, not a human ideal answer. A test case needs only: input (query), actual_output (what the system generated), retrieval_context (chunks actually retrieved). No expected-answer labels. So it's just representative queries + your system's real runtime outputs.

  • Stratify for coverage — include hard/compound/ambiguous cases; hallucination concentrates there, so an all-easy set reports a falsely rosy number.
  • Include "context doesn't contain the answer" cases — highest value; check the model abstains rather than confabulates.
  • Use real retrieved contexts, not idealized ones.
  • Always pair with answer relevancy — faithfulness alone is gameable (answer "I can't determine this" to everything → trivially faithful). High faithfulness + high relevancy is the target.

SECTION 20Golden sets, LLM judges & the labeling pyramid

To compute any of these metrics you need relevance/quality labels. Three sources, a pyramid where each tier checks the one below:

  1. Human annotation (gold standard / anchor) — ~100–300 representative queries graded on your rubric (e.g. 3=perfect, 2=relevant, 1=related, 0=irrelevant, with examples). Trusted, but doesn't scale. You author the rubric — relevance is operationally defined, not handed to you.
  2. LLM-as-judge (scale) — a strong model grades against the same rubric, scaling to thousands of queries. Must be calibrated against the human core (~20–30 cases), and watch for position/length/self-preference bias. The reference-free design (esp. for recall/faithfulness) is what lets a judge work at all.
  3. Implicit production signals (free, noisy) — clicks, dwell, accepted answers, 👍/👎, rephrasings. Massive scale but heavy position bias — needs debiasing.

Why LLM judges over human labels?

Scale: human labeling can't run as an automated regression gate on every change. But human labels are more trustworthy in specialized domains (an LLM judge may miss that two near-identical clauses differ decisively). The mature answer is both: human core to calibrate, LLM judge to extend coverage — each covering the other's weakness.

The operating loop

golden set (human-labeled, frozen)
  ├─ offline: nDCG@k / precision gate on every change
  │            (beat retriever-only baseline, no regression)
  ├─ LLM judge (calibrated) scales to 1000s of queries
  └─ ship behind A/B → online: acceptance, 👍, rephrase rate
            └─ log implicit signals → grow next golden set → repeat
Two truths(1) Evaluate at the k that reaches the user — nDCG@5 if the LLM gets 5 chunks; nDCG@100 is academic. (2) Better retrieval metrics ≠ better answers — track the retrieval metric and the end-to-end answer metric; trust the second when they disagree (a reranker can lift nDCG without moving answer quality if retrieval wasn't the bottleneck).

SECTION 21 — SYSTEMS & FRONTIERDocument loaders

Ingestion quality caps everything downstream. Key gotchas from the LangChain loader taxonomy:

  • .md / Markdown loader — preserves headers as metadata; use it (great for sectioned docs → structure-aware chunking + better citations).
  • .pdf text-based — PyPDFLoader; tables get linearized (the big risk, see §22).
  • .pdf scanned — Unstructured + Tesseract OCR; slow, OCR errors compound.
  • .pdf complex layout — Docling (IBM) is the 2026 open-source winner for tables; LlamaParse paid.
  • HTML/web — strip nav/ads or they pollute every chunk.
  • .docx — tracked changes can leak.
  • Code — language-specific splitter (RecursiveCharacterTextSplitter.from_language).

When converting HTML→Markdown, tables become detectable pipe-delimited blocks (| col | col |), so you can isolate them programmatically.

SECTION 22Handling tables in RAG

Why plain text/Markdown loaders fail tables: they linearize a grid into a token stream, destroying row/column structure. Azure 15% $25,000 M365 10% $10,000 — the model can attach "15%" to the wrong product. In pricing/policy data that's a correctness-critical failure, and it happens at ingestion, before any reranker can help.

Table summarization for retrieval (multi-vector pattern)

The strong technique: extract each table as its own chunk, generate a natural-language summary with a small LLM, and index the summary for retrieval. A recognized best practice (LangChain MultiVectorRetriever) — the "index a summary, retrieve the original" idea.

  • Why: a raw grid of numbers embeds terribly (no semantic signal); a NL summary embeds richly and matches the query's phrasing. Decouples the match-friendly representation from the data-holding one — same parent-child philosophy.
  • Small LLM for summarization is fine on cost (bounded simple task).
The crux questionWhat is passed to generation — the summary or the original table? Index the summary for retrieval, but pass the ORIGINAL table to the LLM for answering. A summary is a lossy compression; answering a precise threshold question from a paraphrase reintroduces the error you were avoiding. Retrieve on the summary, ground on the real numbers.

SECTION 23How citations actually work

Not magic — engineered provenance + a forced reference. Two halves:

Half 1 — Provenance tracking (system, deterministic)

Every chunk carries its metadata from creation: chunk_id, text, embedding, {source, url, page, section, effective_date}. Retrieval returns these full records, so the system already knows "this chunk came from page 14." A foreign key following the data — no intelligence needed. This is why metadata storage matters.

Half 2 — Getting the model to cite (prompt + parse)

  • A — labeled context: inject chunks with IDs ([1] (source, p.14): ...), tell the model to cite the number; parse the [1] out and resolve to the real source. The model produces a pointer; your code resolves it.
  • B — structured output: model returns JSON {answer, citations:[{claim, source_id}]}. Cleaner to parse; the production default.
  • C — post-hoc attribution: model writes answer with no citations, a second pass attributes each sentence to its supporting chunk (similarity or LLM-judge). Independent verification.
The honest catchWith A/B the model chooses which label to attach — and can attach the wrong one. The metadata is always correct (deterministic); which chunk the model points at is a fallible generation. Rigorous systems add a verification layer (does chunk [1] actually entail the claim?). Mental model: citation = deterministic provenance (always correct) + model's pointer choice (fallible) + parse-resolve (deterministic). The metadata never lies; the pointer choice is the only soft link.

SECTION 24RAG-aware training

Modern models are explicitly trained to use injected context better, not just rely on parametric memory:

  • SFT on RAG-shaped data — examples of (query + context → grounded answer); learns the behavior of answering from passages (RankRAG, RA-DIT).
  • Teaching the refusal case — negative examples where context lacks the answer, target = "the context doesn't answer this." Trains the "say I don't know" behavior into weights.
  • Preference alignment (DPO/RLHF) — faithful-to-context responses preferred over parametric-drift/hallucination.
The subtle target"Context as absolute ground truth" is deliberately not the goal — that makes the model trivially manipulable by bad retrieval / prompt injection and unable to flag wrong/stale context. The real target is faithful use of context while retaining judgment (the open "knowledge conflict" problem: what when context contradicts what the model knows?). It reduces but doesn't eliminate hallucination — which is why an external grounding/verification layer is still worth keeping.

SECTION 25Redis & caching

REmote DIctionary Server — an in-memory key-value store: a fast, shared, networked dictionary. Two defining properties:

  • In-memory → microsecond reads (no disk seek); RAM-limited and (by default) volatile → for data you want fast and can afford to lose/regenerate.
  • A server → many app instances hit one shared store, persistent across restarts (beats an in-process dict that dies with the process and isn't shared).

Complexity: easy to use (r.set(k, v, ex=3600) — the ex TTL auto-expiry is 80% of caching value), one-command setup (docker run redis), no schema. The "complicated" reputation is production ops (clustering, replication, failover, eviction tuning), not using it.

When you need it (AI/RAG)

  • Semantic / LLM response caching (the big one) — cache answers to repeated questions. Exact-match (same query string) or semantic (embed query, return a cached answer if similar enough). Cuts LLM cost + latency on hot queries.
  • Embedding cache — text → embedding, avoid re-embedding.
  • Rate limiting / session state / general app caching — why it's in every JD.
The hard part: invalidationCaching is an optimization you add when a specific repeated-cost shows up — not "because best practice." And in a policy/correctness domain, a stale cached answer = serving a wrong (superseded) rule. So caching must include TTL + invalidation tied to document updates. Caching the right thing is easy; knowing when to throw the cache away is the senior signal.

SECTION 26Web search & deep research as RAG

Web search is a (mildly agentic, live) RAG; deep research is a fully agentic multi-hop RAG. The skeleton is identical; every corpus assumption changes.

Closed-corpus RAGWeb search
Corpus / indexyour docs, you build the indexthe internet; index outsourced to a search engine
Retrievalyour hybrid BM25+vectora search API call (Bing/Google/Exa/Tavily)
Trustcurated, trustedunknown, adversarial

Web-search flow

(1) LLM formulates a query → (2) search API returns ranked URLs+snippets → (3) fetch promising pages, strip HTML to text → (4) inject as context → (5) generate with URL citations. Often still single-shot.

Deep research

Plan → decompose into sub-questions → iterative search loops (search, read, decide next search from what was learned) → read full pages → synthesize a long cited report → self-direct until done. = web RAG + agentic planning + multi-hop + long-form synthesis. Minutes and many calls = the agentic tax, paid deliberately.

The trust problem (the core, unsolved issue)

In closed-corpus RAG, retrieving the right chunk means it's correct by assumption. On the open web, retrieval succeeding tells you nothing about truth. Two separate failures: retrieval failure vs. source corruption (found perfectly, but wrong/biased/satire/propaganda/SEO-spam/fabricated).

  • Unknown source reliability — paper, Reddit comment, content farm all return as "results." Mitigated by domain authority, cross-corroboration, recency — heuristics, not guarantees.
  • SEO manipulation / adversarial content — the web is adversarially optimized; AI-generated slop now floods it (slop citing slop).
  • Indirect prompt injection — a fetched page can contain "ignore previous instructions…"; the agent reads attacker-controlled text and may act on it. Larger attack surface the more autonomous the agent. Not solved — mitigations: treat fetched content as untrusted, instruction/data separation, sandboxing.
  • Temporal conflicts — 2019 and 2026 answers both live; recency reasoning is error-prone.
  • Citation ≠ verification — a citation proves "claim came from this page," not "page is correct." A confidently-cited wrong source manufactures false trust.
Interview-grade insight"In closed-corpus RAG my job ends at retrieval quality — the corpus is trusted. Web search breaks that: retrieval can succeed perfectly while the source is wrong or adversarial. So open-web systems need a credibility layer on top of the relevance layer — source authority, cross-corroboration, recency — plus defenses against indirect prompt injection from fetched content. Even then it's mitigation, not a solution, which is why these systems hedge and cite rather than assert." The sharpest danger is the intersection: a page both SEO-optimized to rank and carrying prompt injection — it ranks, gets trusted, then manipulates the agent.

SECTION 27Long-context vs. RAG (quality, cost aside)

The naive intuition — "attention attends to everything, so stuffing the whole doc is better" — is half right. Long-context is better up to a point, then gets worse, and the crossover is well before a million tokens.

Context rot — the finding that breaks the intuition

Attention can technically reach every token (architecturally true) but performance degrades as context grows (behaviorally false that it's used equally). Two different tasks diverge:

  • Needle-in-haystack (retrieve one fact) — models are excellent now; this is what "improved" and what people cite. Intuition holds here.
  • Reasoning over full content (synthesize, compare distant parts) — degrades with length far faster. Finding a fact ≠ holding the whole haystack in working memory. "Lost in the middle" is reduced for retrieval, sticky for reasoning.

Why stuffing the full doc can hurt quality

  • Distractor dilution — burying 3 relevant paragraphs in 1,000 pages forces the model to suppress 997 pages of plausible distractors; it can latch onto related-but-wrong text. RAG's filtering is itself a quality feature.
  • Attention spread thin — a finite soft-weight budget diluted across a million tokens vs. concentrated on 5 chunks.
  • Position effects persist (reduced, not gone) — the "middle" of a million tokens is enormous.

When full-context genuinely wins on quality

  • Whole-document synthesis on a bounded doc ("summarize this contract & find contradictions") — RAG fragments structure and misses cross-references.
  • Chunking would sever critical structure (a clause + its definition 40 pages away).
  • Can't predict what's relevant — giving everything avoids the retrieval-miss failure.
The decision ruleIt's not whether the doc fits — it's what fraction is relevant + lookup vs synthesis. High relevant-fraction or whole-doc reasoning → full-context. Low relevant-fraction over a large corpus → RAG wins on quality (filtering removes distractors). The 2026 answer is composition: RAG to drop the irrelevant 95%, then long-context reasoning over the relevant 5% kept whole. Long context also lets RAG use bigger parent chunks. Caveat: advertised window size is partly marketing — usable-quality length is shorter than the headline, so treat "1M tokens" as an upper bound on retrieval, not a guarantee of reasoning.

SECTION 28What downloads from Hugging Face (and why no optimizer)

A published HF model is an inference package. The files:

  • model.safetensors — the actual trained weights (the only file with the network). .safetensors = safe format (no arbitrary-code execution unlike old pickle .bin).
  • config.json — architecture blueprint (layers, hidden size, heads). Builds the empty skeleton that weights pour into.
  • Tokenizer (several files): vocab.txt (the vocabulary), tokenizer.json (fast Rust tokenizer), tokenizer_config.json (settings), special_tokens_map.json ([CLS],[SEP],[PAD],[MASK]). Multiple files = belt-and-suspenders for fast/slow loaders.
  • README.md — model card, docs only, not used at runtime.

Essential trio for inference: config.json + model.safetensors + tokenizer files. Skip README and format duplicates.

Why no optimizer (the insight)Optimizer state (Adam's momentum + variance, ~2× model size) exists only to compute the next weight update — it's training scaffolding, disposable once final weights exist. Inference is just a forward pass; you don't take gradient steps. So it's stripped from published models. You'd only see it in a training checkpoint saved to resume training (optimizer + LR scheduler + RNG state) — kept locally, not on the Hub. Same train-vs-infer split as "what's needed when."
