Section 01

The confusion-matrix foundation

Every classification metric is a single ratio pulled from one 2×2 table. The metrics split into two camps based on which direction they read that table. Internalize this and precision/recall confusion ends permanently.

The two reading directions

The same TP cell is shared, but the denominator is the whole point:

Recall — fix the truth, ask how much you found
$$\text{Recall} = \frac{TP}{TP + FN} \qquad (\text{the actual-positive row})$$
Precision — fix your alarms, ask how many were right
$$\text{Precision} = \frac{TP}{TP + FP} \qquad (\text{the predicted-positive column})$$
The mnemonic that ends the confusion Recall lives in the row, precision lives in the column. Recall = "of the Real positives" (R-R). Precision = "of my Predictions" (P-P). You can only hurt recall by missing real positives (FN); you can only hurt precision by raising false alarms (FP).

Why they trade off

There is one knob: the decision threshold on the score. Slide it down → predict positive more aggressively → catch more real positives (recall ↑) but fire more false alarms (precision ↓). They move in opposition by construction. That is why a single balancing number (F1) exists and why threshold-free summaries (AUC) exist.

The split that organizes the whole family

SpeciesMetricsWhat it judges
Threshold-dependent
(one frozen cutoff)		accuracy, precision, recall, F1, specificity, MCCPoint estimates from one frozen confusion matrix
Threshold-independent
(sweep & summarize)		AUC-ROC, AUC-PR, log lossRanking quality / calibration of the raw scores, before any threshold
The "95% accuracy" trap Accuracy is a single frozen point that hides both the threshold choice and the class balance. If 99% of emails are not spam, predicting "not spam" for everything scores 99% and catches zero spam. The correct interview reflex: "it depends on the class distribution."
The asymmetry to remember Precision, recall, and F1 never touch TN. That is their blind spot under imbalance. MCC is the only basic metric using all four cells, which is why it is the most honest single number when classes are skewed.
Section 02

AUC-ROC: the threshold sweep

A classifier outputs a score, not a decision. Turning a score into a label needs a threshold, and every threshold gives a different confusion matrix. ROC sidesteps "which threshold?" by asking: across all thresholds, how well do my scores separate positives from negatives? It judges the ranking, not any one cutoff.

The two axes

$$\text{TPR} = \text{Recall} = \frac{TP}{TP+FN} \quad (y\text{-axis})$$ $$\text{FPR} = \frac{FP}{FP+TN} = 1 - \text{Specificity} \quad (x\text{-axis})$$

The symmetry: TPR reads across the positive row, FPR across the negative row. Both are "of the actual class, how much did I flag positive" — you want one high, the other low.

How the curve is actually plotted

This is the mechanical procedure people skip. Forget the integral:

  1. Score every example → list of (score, label) pairs.
  2. Sort by score, highest first.
  3. Start threshold at +∞: nothing predicted positive → point (0,0).
  4. Lower the threshold one example at a time. Each crossed example flips to "predicted positive":
    • actually positive → TP up → step UP (TPR increases)
    • actually negative → FP up → step RIGHT (FPR increases)
  5. Threshold below every score: everything positive → (1,1).

So ROC is a staircase: up for every positive, right for every negative. A perfect model scores all positives above all negatives → straight up to (0,1), then right — hugging the top-left corner.

The math — what AUC actually equals

The integral form \( \text{AUC} = \int_0^1 \text{TPR}\,d(\text{FPR}) \) is correct but least useful. The interpretation that wins interviews:

AUC = probability a random positive outranks a random negative
$$\text{AUC} = P(s^+ > s^-) = \frac{1}{P \cdot N}\sum_{i \in \text{pos}}\sum_{j \in \text{neg}} \mathbb{1}[s_i > s_j]$$

The double sum counts every (positive, negative) pair where the positive outscored the negative, divided by the total number of pairs \(P\cdot N\). The area under the staircase is this count — each up-step (a positive) sweeps across the fraction of negatives already ranked below it. Area is just the geometric shadow of "how often positives beat negatives."

Depth signal AUC is a normalized Mann-Whitney U statistic (equivalently the Wilcoxon rank-sum). Dropping this connection signals real understanding.

Reading the key values

  • AUC = 1.0 — every positive outscores every negative; perfect ranking.
  • AUC = 0.5 — coin flip; the diagonal. Means random, not "wrong."
  • AUC = 0.3 — not bad, it's informative with flipped labels; invert predictions → 0.7.
  • Threshold-free — measures whether scores rank correctly, independent of cutoff. Its superpower, and (under imbalance) its weakness.
60-second interview answer

"ROC plots true positive rate against false positive rate as you sweep the threshold across every value. AUC is the area under it, and the cleanest interpretation is probabilistic: the chance a random positive is scored higher than a random negative. So 1.0 is perfect ranking, 0.5 is random. It's threshold-independent, great for comparing models before committing to a cutoff, but misleadingly optimistic under heavy class imbalance because FPR's denominator is dominated by true negatives — there I'd switch to AUC-PR."

Section 03

AUC-PR: telling the truth under imbalance

Same machinery as ROC — sweep the threshold, plot two quantities — but one axis is swapped, and that single swap changes everything when positives are rare.

Same sweep, different axes

$$\text{Precision} = \frac{TP}{TP+FP} \quad (y\text{-axis}) \qquad \text{Recall} = \frac{TP}{TP+FN} \quad (x\text{-axis})$$

Why the curve behaves so differently

Look at the denominators — this is the entire story:

  • ROC's x-axis (FPR) has denominator FP + TN — both negative-class cells. TN is enormous when negatives dominate.
  • PR's y-axis (precision) has denominator TP + FP — it never touches TN.

In a fraud problem with 99.9% legitimate transactions, the giant pile of correctly-ignored legit transactions (TN) inflates ROC but is invisible to PR. PR only looks at the positive class and your alarms about it — it can't hide behind a mountain of easy negatives.

Structural difference ROC always runs (0,0)→(1,1) monotonically — a clean staircase. The PR curve is not monotonic: as the threshold drops, recall only increases, but precision jiggles — up when you cross a real positive, down when you cross a false positive. PR sawtooths downward.

The baseline — the most testable contrast

  • ROC random baseline = the diagonal, AUC = 0.5, always, regardless of class balance.
  • PR random baseline = a horizontal line at precision = positive prevalence. If positives are 0.1% of data, random AUC-PR ≈ 0.001.

Consequence: AUC-PR is not comparable across datasets with different prevalence. An AUC-PR of 0.4 is fantastic against a 0.001 baseline, garbage against 0.5. ROC's fixed 0.5 baseline is comparable across datasets but blind to imbalance. Pick your poison.

Why ROC lies under imbalance — concretely

1,000,000 transactions, 1,000 fraud (0.1%). Model flags 2,000, catches 900 real ones.

  • Recall = 900/1000 = 0.90 — looks great
  • FPR = 1100/999000 ≈ 0.0011 — looks incredible; ROC sits top-left, AUC superb
  • Precision = 900/2000 = 0.45 — fewer than half your fraud alarms are real

ROC's FPR was diluted to near-zero by the 999,000-strong TN denominator. Precision, ignoring TN, exposed the truth. That is the whole reason AUC-PR exists.

The math behind AUC-PR

Unlike AUC-ROC, there is no clean probabilistic interpretation. It is the area under precision-as-a-function-of-recall:

$$\text{AUC-PR} = \int_0^1 \text{Precision}(r)\,dr$$

Computed two ways — and the distinction is a trap:

  1. Trapezoidal (auc(recall, precision)) — connects points with straight lines. Known to be optimistic: linear interpolation in PR space is not achievable by any real classifier.
  2. Average Precision (AP) — the right way, what average_precision_score computes:
$$\text{AP} = \sum_n (R_n - R_{n-1})\cdot P_n$$

A weighted mean of precisions, weighted by the increase in recall at each threshold. Avoids the optimistic interpolation. "I use average_precision_score, not trapezoidal AUC, because trapezoidal interpolation in PR space is optimistic" is a strong signal.

Sharper decision rule than "<5% positive" The test isn't just "is it imbalanced," it's do I care about true negatives at all? If the negatives are a vast uninteresting background (fraud, rare disease, retrieval, anomaly detection) and all your value is in the positive class — use PR. If you genuinely care about correctly clearing negatives — ROC is meaningful.
60-second interview answer

"The PR curve plots precision against recall as you sweep the threshold — same sweep as ROC, but precision replaces FPR. The key difference is precision's denominator is TP + FP, so it never touches true negatives, while ROC's FPR is dominated by the true-negative count under imbalance. That's why ROC looks optimistically great on rare-positive problems while PR stays honest. The PR random baseline is the positive prevalence, not 0.5, so AUC-PR isn't comparable across datasets with different base rates. In practice I use average precision rather than trapezoidal AUC because linear interpolation in PR space is optimistic. Rule of thumb: if I don't care about true negatives — fraud, anomaly, retrieval — I report PR."

Section 04

Pure-Python: plotting both curves

Everything built from the confusion matrix up, no sklearn, threshold swept in 0.01 steps with an explicit for loop. On the demo data this prints AUC-ROC ≈ 0.84 (looks strong) but AUC-PR ≈ 0.58 against a 0.19 baseline (sober) — the imbalance optimism, right there in the numbers.

Interview point about the 0.01 grid The 0.01-step sweep is the practical approximation. Libraries instead use every unique predicted score as a threshold, so the curve passes exactly through every real operating point. The fixed grid can skip or smear points when scores cluster tightly. Knowing why the grid is an approximation is the depth signal. 
import matplotlib.pyplot as plt
import random

# 0. Fake data: probabilities + true binary labels (~20% positive).
def make_data(n=400, pos_rate=0.2, seed=42):
    random.seed(seed)
    probs, labels = [], []
    for _ in range(n):
        y = 1 if random.random() < pos_rate else 0
        if y == 1:
            p = min(1.0, max(0.0, random.gauss(0.65, 0.18)))  # positives score higher
        else:
            p = min(1.0, max(0.0, random.gauss(0.35, 0.18)))  # with overlap (realistic)
        probs.append(p); labels.append(y)
    return probs, labels

# 1. Confusion matrix at a threshold. Predict positive if prob >= threshold.
def confusion_matrix(probs, labels, threshold):
    tp = fp = tn = fn = 0
    for p, y in zip(probs, labels):
        pred = 1 if p >= threshold else 0
        if   pred == 1 and y == 1: tp += 1
        elif pred == 1 and y == 0: fp += 1
        elif pred == 0 and y == 0: tn += 1
        else:                      fn += 1
    return tp, fp, tn, fn

# 2. Rates. Guard every division by zero explicitly.
def tpr(tp, fn):  return tp / (tp + fn) if (tp + fn) else 0.0   # recall (ROC y, PR x)
def fpr(fp, tn):  return fp / (fp + tn) if (fp + tn) else 0.0   # fall-out (ROC x)
def precision(tp, fp): return tp / (tp + fp) if (tp + fp) else 1.0  # no preds => 1.0

# 3. Sweep threshold 1.01 -> -0.01 in 0.01 steps so curve spans (0,0)->(1,1).
#    NOTE: production keys on sorted(set(probs), reverse=True) instead of a grid.
def sweep(probs, labels, step=0.01):
    roc_pts, pr_pts = [], []
    t = 1.01
    while t >= -0.001:
        tp, fp, tn, fn = confusion_matrix(probs, labels, t)
        roc_pts.append((fpr(fp, tn), tpr(tp, fn)))       # (x=FPR,    y=TPR)
        pr_pts.append((tpr(tp, fn), precision(tp, fp)))  # (x=Recall, y=Precision)
        t -= step
    return roc_pts, pr_pts

# 4. Trapezoidal area. Sort by x first (PR is non-monotonic).
def auc_trapezoid(points):
    pts = sorted(points, key=lambda p: p[0])
    area = 0.0
    for i in range(1, len(pts)):
        x0, y0 = pts[i-1]; x1, y1 = pts[i]
        area += (x1 - x0) * (y0 + y1) / 2.0   # width * average height
    return area

# 5. Run + plot.
probs, labels = make_data()
P = sum(labels); baseline = P / len(labels)   # PR baseline = positive prevalence
roc_pts, pr_pts = sweep(probs, labels, step=0.01)
auc_roc = auc_trapezoid(roc_pts)
auc_pr  = auc_trapezoid(pr_pts)
print(f"AUC-ROC = {auc_roc:.4f}")
print(f"AUC-PR  = {auc_pr:.4f}  (random baseline = {baseline:.3f})")

fig, (axL, axR) = plt.subplots(1, 2, figsize=(12, 5))
rx, ry = zip(*roc_pts)
axL.plot(rx, ry, lw=2, label=f"ROC (AUC={auc_roc:.3f})")
axL.plot([0,1],[0,1],"--",label="random (0.5)")
axL.set_xlabel("FPR"); axL.set_ylabel("TPR"); axL.set_title("ROC"); axL.legend()
px, py = zip(*pr_pts)
axR.plot(px, py, lw=2, label=f"PR (AUC={auc_pr:.3f})")
axR.axhline(baseline, ls="--", label=f"random ({baseline:.2f})")
axR.set_xlabel("Recall"); axR.set_ylabel("Precision"); axR.set_title("PR"); axR.legend()
plt.tight_layout(); plt.show()

Three things an interviewer probes — all visible here

  • The precision 0/0 edge case. At the top of the sweep nothing is predicted positive, so precision is undefined. Convention here: 1.0, which creates the precision spike at the left edge. sklearn sets the first point to precision=1, recall=0.
  • The sort before trapezoid. ROC points come out sorted by FPR; PR points need sorting by recall. And even sorted, trapezoidal PR area is the optimistic estimate — the honest metric is Average Precision.
  • Grid vs unique scores. The 0.01 grid is fine for evenly-spread scores; it skips or revisits operating points when scores pile up in a narrow band. Production keys on unique scores.
Section 05

R² and Adjusted R²

Core intuition: R² measures how much better your model is than just predicting the mean. The dumbest model ignores all features and always predicts \(\bar{y}\).

TSS — error of the mean-model (the variance you're stuck with)
$$\text{TSS} = \sum_i (y_i - \bar{y})^2$$
RSS — error of your actual model
$$\text{RSS} = \sum_i (y_i - \hat{y}_i)^2$$
R² — fraction of the mean-model's error you eliminated
$$R^2 = 1 - \frac{\text{RSS}}{\text{TSS}} = 1 - \frac{\sum (y_i - \hat{y}_i)^2}{\sum (y_i - \bar{y})^2}$$

Read it plainly: of the variance the mean-model left on the table, what fraction did my model remove? RSS/TSS is the error you failed to remove; 1 minus that is the error you did remove.

  • R² = 1 — RSS = 0, perfect.
  • R² = 0 — exactly as good as predicting the mean.
  • R² < 0 — worse than the mean. R² is not bounded in [0,1]; on test data or a mis-specified model, RSS can exceed TSS.

Is R² like accuracy? Half right.

Valid parallel: both are normalized, unitless, higher-is-better scores with an interpretable ceiling, so both compare across problems without worrying about units. RMSE of "2.3" means nothing without scale; R² of 0.92 is interpretable alone. R² is the scale-free regression metric the way accuracy is the scale-free classification metric.

Where it breaks: accuracy has a fixed floor (base rate) and ceiling (1.0). R²'s baseline is your own dataset's variance, so R² is relative to how hard your specific problem is:

  • High R² can be trivial — tiny target variance, or autocorrelated time-series data.
  • Low R² can be excellent — in noisy domains (individual stock returns), R² = 0.05 may be genuinely valuable.

R² vs the other regression metrics

Adjusted R² — and the real reason it exists

The core problem R² can never decrease when you add a feature, even pure noise. OLS finds some tiny coefficient that fits the residuals, RSS drops slightly, R² ticks up. So raw R² rewards complexity for free and is useless for comparing models with different feature counts.
Adjusted R² — penalize per degree of freedom
$$R^2_{\text{adj}} = 1 - \frac{\text{RSS}/(n-p-1)}{\text{TSS}/(n-1)} = 1 - (1-R^2)\,\frac{n-1}{n-p-1}$$

The mechanism: compare sums of squares per degree of freedom. Add a feature → \(p\) rises → \((n-p-1)\) shrinks → the penalized RSS term inflates. A new feature only raises adjusted R² if it cuts RSS more than enough to overcome that shrinking-degrees-of-freedom penalty. A noise feature can't pay for itself, so adjusted R² drops.

  • Adjusted R² can decrease and go negative, unlike how people imagine R².
  • The \((n-1)/(n-p-1)\) factor is a poor-man's out-of-sample correction — the grain of truth behind the "sampling/generalization" instinct.
  • It's a model-selection tool, not a reporting metric. Use it to pick features; still report plain R² (or RMSE) for final quality. Using R² to pick features would always select the most complex model.
60-second interview answer

"R² is one minus RSS over TSS — the fraction of the target's variance the model explains relative to predicting the mean. R² = 1 is perfect, 0 is no better than the mean, and it can go negative when the model is worse than the mean. Its strength is being scale-free, so unlike RMSE it's comparable across problems. Its weakness is that the baseline is the dataset's own variance, so high R² can be trivial on low-variance or time-series data and low R² can still be great in noisy domains. The big gotcha: R² never decreases when you add features, even useless ones, so for comparing models of different complexity you use adjusted R², which penalizes the feature count by dividing sums of squares per degree of freedom. In practice I report RMSE for error magnitude and R² for explained variance."

Section 06

NDCG: position-aware ranking

The name is the recipe read backwards: Gain → Cumulative Gain → Discounted Cumulative Gain → Normalized DCG. Each word adds one idea.

Layer 1 — Gain

Gain is the relevance of one item. The key shift from classification: relevance is graded (a 5-star result has more gain than a 3-star), not binary. That's the first thing distinguishing NDCG from MAP.

Layer 2 — Cumulative Gain (CG)

$$\text{CG@K} = \sum_{i=1}^{K} rel_i$$

Fatal flaw: CG is position-blind. [3,2,0] and [0,2,3] have identical CG, yet the first is far better. We need to punish good items for sitting low.

Layer 3 — Discounted Cumulative Gain (DCG) — the position-aware part

$$\text{DCG@K} = \sum_{i=1}^{K} \frac{rel_i}{\log_2(i+1)}$$

Why \(\log_2(i+1)\)? Walk the discounts: position 1 → 1/log₂2 = 1.00; position 2 → 0.63; position 3 → 0.50; position 10 → 0.29. The log makes the decay gentle: the 1→2 drop is big, the 9→10 drop is tiny — matching how user attention actually falls off. A linear 1/i discount would punish lower ranks too harshly.

Common alternative (used by most libraries and the original web-search papers), weighting high relevance exponentially:

$$\text{DCG@K} = \sum_{i=1}^{K}\frac{2^{rel_i}-1}{\log_2(i+1)}$$

Layer 4 — Normalize → NDCG

DCG is unbounded and incomparable across queries (a query with more relevant docs racks up more gain regardless of ranking quality). Divide by the best possible DCG for that query:

$$\text{NDCG@K} = \frac{\text{DCG@K}}{\text{IDCG@K}}$$

IDCG = the DCG of the ideal ordering (all items sorted by relevance, best first) — the ceiling for that query. Now bounded in [0,1], and comparable across queries because each is normalized against its own ideal.

60-second interview answer

"DCG sums each item's relevance discounted by a log of its position, so relevant items ranked higher count more — the log makes the decay gentle, matching how user attention falls off. NDCG normalizes DCG by the ideal DCG, the score of a perfect relevance-sorted ranking, putting it in [0,1] and making it comparable across queries of different difficulty. I use it when relevance is graded."

Section 07

Precision@k, AP, and MAP

These form a ladder, and the confusion comes from collapsing its rungs. The word "average" appears twice in "mean average precision" for a reason.

Rung 1 — Precision@k (one cutoff, one query)

$$\text{Precision@k} = \frac{\text{relevant items in top } k}{k}$$

For list [✓, ✗, ✓, ✗, ✓]: P@1 = 1.0, P@3 = 2/3 = 0.67, P@5 = 0.60. One list, one cutoff, one number.

The limitation that the next rung fixes Precision@k is position-blind within the top k. [✓,✓,✗] and [✗,✓,✓] both score P@3 = 0.67 — it can't see that the first front-loaded the relevant items.

Rung 2 — Average Precision (AP) (one full ranked list)

AP is not precision at one k. It is the average of precision@k values, computed only at the positions where a relevant item appears, for a single query:

$$\text{AP} = \frac{1}{R}\sum_{k=1}^{n} \text{Precision@k}\cdot rel(k)$$

where \(R\) = total relevant items and \(rel(k)=1\) only if position \(k\) holds a relevant item (so you "sample" precision only at the hit positions).

Walk-through on [✓, ✗, ✓, ✗, ✓], R = 3

  • pos 1 ✓ → P@1 = 1/1 = 1.00
  • pos 2 ✗ → skip
  • pos 3 ✓ → P@3 = 2/3 = 0.67
  • pos 4 ✗ → skip
  • pos 5 ✓ → P@5 = 3/5 = 0.60
$$\text{AP} = \frac{1.00 + 0.67 + 0.60}{3} = 0.756$$
The sentence that resolves the confusion Each time you hit a relevant item, AP takes a snapshot of how clean the list has been up to that point. A relevant item near the top is measured while precision is still high → big contribution. One buried at the bottom is measured after junk piles up → small contribution. So AP rewards putting relevant items early — exactly the position-awareness precision@k lacks. Precision@k is a snapshot at one position; AP is the average of those snapshots taken at every relevant hit.

Rung 3 — Mean Average Precision (MAP) (across queries)

$$\text{MAP} = \frac{1}{|Q|}\sum_{q \in Q} \text{AP}_q$$

The full ladder by name:

MetricScopePosition-aware?
Precision@kone cutoff, one queryNo — blind inside top k
AP (or AP@k)one query, all relevant hitsYes — via precision at hits
MAPaveraged across all queriesYes
Why "average" twice MAP = mean (over queries) of average (over relevant positions) precision. Two stacked averages. Confusing MAP with precision@k means collapsing those two averaging steps.

NDCG vs MAP — when to use which

Deciding question: is your relevance binary or graded?

MAPNDCG
RelevanceBinary (relevant / not)Graded (0–5, stars)

Natural homeDocument retrieval, yes/no relevanceRecommendations, learning-to-rank with quality grades

Third sibling — MRR (Mean Reciprocal Rank): when only the rank of the first relevant hit matters (Q&A, "I'm feeling lucky"), \( \text{MRR}=\frac{1}{|Q|}\sum \frac{1}{\text{rank}_i} \). It ignores everything after the first success.

60-second interview answer

"Precision@k is the fraction of the top k that's relevant — one number, blind to ordering inside the top k. Average precision averages the precision@k values taken only at the ranks where a relevant item appears, so it rewards putting relevant items early; that's the position-awareness precision@k lacks. MAP is the mean of average precision across all queries — so it's two averages, over relevant positions within a query, then over queries. I use MAP for binary relevance and NDCG when relevance is graded."

Section 08

Perplexity as effective branching

Beyond "exponentiated average negative log-likelihood." Perplexity is a quantity living in the space of probability distributions, measured in effective vocabulary size, that happens to equal the exponential of a cross-entropy you're already minimizing.

The one-sentence reframe Perplexity is the effective branching factor of the model's predictive distribution. Perplexity 20 means that, averaged over every position, the model is as uncertain as if choosing uniformly among 20 equally-likely next tokens. The word "effective" is the whole concept: not literally 20 choices, but the uniform distribution that would leave you equally confused.

Building it from entropy

Entropy is average surprise; for a uniform distribution over \(k\) outcomes, \(H = \log k\). Invert it: a distribution with entropy \(H\) (in bits) is as uncertain as a uniform distribution over

$$k = 2^{H}$$

This \(k\) is the perplexity of the distribution. Perplexity is just entropy exponentiated — back into "number of outcomes" units instead of "bits." Entropy lives in log-space (additive, abstract); perplexity in count-space (multiplicative, intuitive). Same information, two coordinate systems.

Subtlety 1 — it's a cross-entropy, not an entropy

You don't know the true distribution \(p\); you have samples (the test text) and your model \(q\). What you compute is the cross-entropy:

$$H(p, q) = -\sum_x p(x)\log q(x) \approx -\frac{1}{N}\sum_{i=1}^{N} \log q(w_i \mid w_{<i})$$

The average NLL is not the definition of perplexity — it's an empirical estimator of the cross-entropy \(H(p,q)\). Perplexity is \(b\) raised to it.

The floor you can never beat By Gibbs' inequality, \(H(p,q) \ge H(p)\), equality iff \(q=p\). Cross-entropy is bounded below by the true entropy of language, so perplexity has a floor: \(2^{H(\text{language})}\). The excess decomposes exactly as KL divergence: $$H(p,q) - H(p) = D_{\text{KL}}(p \,\|\, q)$$ So perplexity measures an irreducible floor set by language itself plus your model's avoidable KL divergence from the truth. Minimizing perplexity drives \(q \to p\).

Subtlety 2 — the geometric-mean structure

By the chain rule \(q(w_1,\dots,w_N) = \prod_i q(w_i \mid w_{<i})\). Perplexity is the inverse geometric mean of per-token probabilities:

$$\text{PPL} = \left(\prod_{i=1}^{N}\frac{1}{q(w_i \mid w_{<i})}\right)^{1/N} = \left(\prod_{i=1}^{N} q(w_i \mid w_{<i})\right)^{-1/N}$$

Each token contributes \(1/q\) — its local branching factor. Geometric (not arithmetic) because probabilities multiply along the sequence, so the natural average is additive in log-space.

The brutal asymmetry Geometric mean is dominated by its smallest terms. One token assigned probability 0.0001 contributes local perplexity 10,000 and drags the whole geometric mean up violently. One confidently-wrong token poisons the perplexity of an entire document. This is why perplexity is so sensitive to tail behavior.

The four caveats that separate a real answer from a recited one

  • Only comparable under identical tokenization. Perplexity is per-token normalized. A 256K-vocab model chops text into fewer tokens than a 32K-vocab one, spreading the joint probability over fewer factors, mechanically lowering per-token perplexity without being "better." Comparable only with identical test set and tokenizer. (Bits-per-byte normalizes by raw bytes to escape this.)
  • Measures distributional fit, not generation. Computed teacher-forced (every position conditions on the true history). It never watches the model generate. A model can have great perplexity yet produce repetitive or degenerate text under sampling.
  • Blind to what you care about in a chat model. Factuality, helpfulness, safety, instruction-following — none captured. Confidently predicting the next token of a factually wrong test sentence gets rewarded. Central for pretraining; near-useless for an aligned assistant.
  • The KL-floor explains diminishing returns. Since perplexity bottoms out at \(2^{H(\text{language})}\), early training crushes huge KL cheaply; late training fights for the last fraction of a bit. The information-theoretic shadow of the scaling-law plateau.
Interview answer, at depth

"Perplexity is the exponential of the cross-entropy between data and model, making it the effective branching factor of the next-token distribution — the size of the uniform distribution that would leave you equally uncertain. Because cross-entropy splits into the entropy of language plus the KL divergence from model to truth, perplexity measures an irreducible floor plus the model's avoidable divergence, so minimizing it drives the distribution toward the true one. Structurally it's the inverse geometric mean of per-token probabilities, so it's dominated by the worst-predicted tokens — one near-zero probability blows it up. The critical caveat: it's per-token normalized, so only comparable under identical tokenization and test set; a bigger vocabulary mechanically lowers it. And it scores teacher-forced fit, not generation, so it says nothing about factuality or sampling quality — central for pretraining, replaced by downstream and human evals for aligned models."

Section 09

SHAP & Shapley values

The rare ML tool with a real theorem behind it. Everything rests on one idea from cooperative game theory.

The fundamental reframe — a prediction is a payout to divide

Shapley (1950s) answered: a team cooperates to produce some value; how do you fairly split it, given players contribute differently and interact? He proved there's exactly one split satisfying a set of fairness axioms. Not "a good way" — the unique way. That uniqueness is why SHAP has weight LIME and permutation importance lack.

Game theorySHAP translation
The gameMaking one prediction for one specific example
The playersThe feature values of that example
The payout\(f(x) - \mathbb{E}[f(x)]\): how far this prediction lands from average
A player's Shapley valueIts fair share of that gap
Why SHAP is natively per-feature, per-example The game is defined for one example (its feature values are the players); the solution splits the payout across features. The math is local by construction. A global explanation is just an aggregate of many local ones.

Contribution is marginal — but order matters

$$\text{marginal contribution of } i \text{ to coalition } S = f(S \cup \{i\}) - f(S)$$

The problem: a feature's marginal contribution depends on who's already in the room (features interact). So there's no single "contribution of income" — it depends on the order features joined. Shapley's fix: average the marginal contribution over every possible order.

The formula, now readable

$$\phi_i = \sum_{S \subseteq N \setminus \{i\}} \frac{|S|!\,(|N|-|S|-1)!}{|N|!}\,\big[f(S \cup \{i\}) - f(S)\big]$$
  • \(f(S\cup\{i\}) - f(S)\) — the marginal contribution (the arrows above).
  • \(\sum_{S \subseteq N\setminus\{i\}}\) — over every coalition not already containing \(i\) (every room \(i\) could enter).
  • \(\frac{|S|!\,(|N|-|S|-1)!}{|N|!}\) — combinatorial weight = fraction of the \(n!\) orderings producing this "join \(S\), then add \(i\)" situation. Exactly the "average over orderings," written compactly.

The four axioms — why this is the answer

  • Efficiency (→ additivity). Contributions sum exactly to \(f(x) - \mathbb{E}[f(x)]\). A complete, exact decomposition with nothing left over — the killer property for explainability and compliance.
  • Symmetry. Features contributing identically to every coalition get equal values. No privilege by name or position.
  • Dummy / Null. A feature contributing 0 to every coalition gets exactly 0.
  • Linearity. Shapley values of an ensemble = sum of per-model values. This makes TreeSHAP work for boosted forests.

The "consistency" property follows: make a feature matter more and its SHAP value can't go down. No competing method satisfies the whole set — SHAP is the provably fair decomposition.

How "present vs absent" actually works

A trained model needs all inputs — you can't literally remove a feature. SHAP defines \(f(S)\) as the expected prediction, marginalizing absent features over a background distribution:

$$f(S) = \mathbb{E}_{x_{\bar{S}}}\big[f(x_S, x_{\bar{S}})\big]$$

Hold present features at this example's values; for absent ones, average the output over plausible values from a background/reference dataset. "Absent" = "replaced by random realistic values and averaged out." This is why SHAP needs a background set, and why the baseline \(\mathbb{E}[f(x)]\) is the prediction when all features are absent.

The deepest honest critique Marginalize using the marginal distribution of absent features (interventional SHAP — breaks correlations) or the conditional distribution given present ones (conditional SHAP — respects correlations but can leak attribution to correlated-but-unused features)? Naming this interventional-vs-observational tension signals real depth.

The cost, and why TreeSHAP matters

Exact computation sums over all \(2^n\) subsets — exponential. Intractable past a handful of features. In practice:

  • KernelSHAP — model-agnostic; samples coalitions, solves a weighted linear regression whose coefficients are the Shapley values. Any model, but slow.
  • TreeSHAP — for tree ensembles, exploits tree structure for exact Shapley values in \(O(TLD^2)\) (trees × leaves × depth²) instead of \(O(2^n)\). Why SHAP exploded in tabular ML.
  • DeepSHAP / GradientSHAP — backprop-style approximations for neural nets.

Local → global

Everything above is local — one example, one set of values. Global views are aggregations:

  • Global importance = mean of \(|\phi_i|\) across examples.
  • Beeswarm/summary plot = every example's \(\phi_i\) per feature — magnitude and direction.
  • Dependence plot = \(\phi_i\) vs the feature's value — nonlinear effects and interactions.

"Per-feature, per-example" isn't a usage choice — it's the native granularity. Global exists only because you can average a fundamentally local quantity.

Interview answer

"SHAP attributes a single prediction to its features using Shapley values from cooperative game theory. The prediction is a payout, the feature values are players, and we fairly divide the gap between this prediction and the average. A feature's contribution is its marginal effect, but that depends on which other features are already considered because features interact, so the Shapley value averages the marginal contribution over all orderings. It's the unique attribution satisfying efficiency, symmetry, dummy, and linearity — efficiency being key, since contributions sum exactly to prediction minus baseline, a complete decomposition. Absent features are marginalized over a background dataset, which is why SHAP needs a reference distribution. Exact computation is exponential, so TreeSHAP computes it exactly in polynomial time for tree ensembles, KernelSHAP approximates for any model. It's natively local — one set of values per example — and global importance is the mean absolute SHAP value across examples."

The reflexes

  • "95% accuracy, good?" → "Depends on class distribution."
  • "Which metric?" → "What's more expensive, a false positive or a false negative?"
  • ROC vs PR → "PR when positives are rare and I don't care about true negatives."
  • AUC meaning → "Probability a random positive outranks a random negative — a Mann-Whitney U."
  • RMSE vs MAE → "MSE targets the mean, MAE the median of the target distribution."
  • R² gotcha → "Never decreases when adding features; use adjusted R² to compare complexity."
  • NDCG vs MAP → "Graded relevance → NDCG; binary → MAP."
